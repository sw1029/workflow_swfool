# Maintain Cycle Ledger

## Overview

Use this skill to record orchestration state as durable workflow evidence under `.task/cycle/<cycle-id>/` and to publish a validated attempt as one authoritative current revision. Ordinary stage events are workflow evidence, not validation proof and not implementation code. A validation result remains a candidate until finalization succeeds and its receipt re-verifies.

Classify each write with this skill's `authority.operations.json` and the shared [authority v2 contract](../manage-agent-authority/references/authority-v2-contract.md). Appending evidence and finalizing authoritative cycle state are separate operations; bind and reserve the exact cycle subject before either mutation.

Use `PYTHONPATH="${CODEX_HOME:-$HOME/.codex}/skills/orchestrate-task-cycle/scripts" python3 -m orchestrate_task_cycle ledger` for deterministic writes whenever possible.

## Domain Adapter Contract

Ledger maintenance may consume these optional adapter hooks only for close hygiene, S10 blocker persistence visibility, S13 terminal-outcome latching, and S14 hook-demand routing:

- `record_retention_policy(**context) -> dict`: O5 helper returning adapter-owned retention classes, hot/warm/cold boundaries, large-packet policy, archive manifest policy, and rotation exclusions. The adapter owns retention boundaries and thresholds. If absent or malformed, fail quiet and perform no rotation.
- `chronic_threshold(blocker_signature, **context) -> int|dict`: optional S10 helper returning the adapter-owned generation threshold for chronic blocker visibility. The ledger stores `first_seen_generation` and `consecutive_generation_count` when blocker-signature evidence is supplied, and may mark `chronic_blocker` debt when the count reaches the threshold. If absent or malformed, fail quiet to existing ledger writes and record `chronic_threshold_unverified` only when chronic status is queried.
- `terminal_outcome_key(terminal_state, **context) -> str|dict`: optional S13 helper returning an opaque terminal outcome key after adapter-owned facet collapse. The ledger may also consume a caller/loopback-supplied key that was collapsed through `facet_root_map`. If absent or malformed, fail quiet to existing ledger writes and record `terminal_outcome_key_unverified` only when quiescence latching is queried.
- `hook_demand_threshold(**context) -> int|dict`: optional S14 threshold for `adapter_hook_demand` decision-relevant skips. The adapter, caller, or registry policy owns the threshold. If absent or malformed, fail quiet to existing ledger writes, keep demand rows warn-only, emit `hook_demand_threshold_unverified` when threshold evaluation was otherwise requested, and do not change any stage status.
- `hook_registry(**context) -> dict`: optional S14 helper returning callable `defined_hooks` and optional opaque source ids for adapter functions. A hook counts as supplied only when it is present as a callable function in this registry or equivalent adapter-load evidence; task-pack strings, metric names, scalar labels, or claimed hook ids do not satisfy wiring. If absent or malformed, fail quiet, emit `hook_registry_unverified` when wiring verification was otherwise requested, and keep hook demand as warning/select-routing evidence.

Do not hardcode retention periods, packet-size thresholds, archive locations, blocker-family meanings, chronic thresholds, terminal outcome meanings, input-delta meanings, hook names, or hook-demand thresholds in this skill. O5 is a hygiene contract only; S10 exists because per-cycle judgments do not otherwise count how many generations the same blocker signature has persisted. S13 is the terminal-outcome specialization of S10 and must not be counted as another blocker family. S14 is the G-ADAPTER demand-ledger extension and is warn/select-routing evidence, not a second gate or hard stop. None of O5, S10, S13, or S14 may change validation, progress, or close verdicts.

## Workflow

1. Initialize the cycle before authority or governance work:
   - Create `.task/cycle/<cycle-id>/initialization.json` and immutable `initialization.provenance.json`, which seals the complete canonical initialization metadata; create `stage.jsonl` only when the first canonical `context` event is appended.
   - Create `.task/cycle/<cycle-id>/current_stage.json`.
   - Create `.task/cycle/<cycle-id>/packets/`.
   - For a new CLI-created protocol-v2 cycle, persist `workflow_contract_profile: compiler_first_enforced_v1`. Do not retrofit provenance or profile markers onto an existing ledger; initialize a new cycle instead.
   - Let the helper generate collision-resistant cycle IDs unless a caller already owns a stable path-safe ID. Reinitializing the same explicit ID is idempotent only when task and reason match.
   - Require the resolved cycle directory, lock, JSONL, snapshot, and dashboard paths to remain under the workspace `.task/cycle` directory. Parent traversal and symlink redirection fail before any write.
2. Treat `init` as storage bootstrap metadata in `initialization.json`, not a canonical stage event. Append `context` as the first canonical `stage.jsonl` row, followed by `authority`, `repo_skill_adapter_scan`, `acceptance`, `route_plan`, `validation_scope_plan`, `validation_set_plan`, `governance`, `result_contract`, `repo_skill_adapter_validate`, `ledger_append`, `code_structure_audit`, `run`, `qualitative_review`, `loopback_audit`, `validation_set_build`, `visible_increment`, `repo_skill_gap_analysis`, `cycle_efficiency_profile`, `validation_scope_finalize`, `index_pre_validate`, `validate`, `issue`, `schema_pre_derive`, `derive`, `schema_post_derive`, `index`, `commit`, `dashboard`, `report`, and `closeout_commit`.
3. Include the minimum event fields when known: `cycle_id`, `event_id`, `step`, `status`, `reason`, `task_id`, `completed_task_id`, `next_task_id`, `changed_files`, `artifacts`, `artifact_refs`, `unchanged_refs`, `validation_verdict`, `progress_verdict`, `blockers`, `authority_policy`, `authority_policy_source`, and `created_at`.
   - Require an explicit non-empty stage `status`. Reject a payload `cycle_id` that differs from the selected ledger. Let the helper generate collision-resistant event IDs, or reuse a caller-owned event ID only for an identical idempotent retry; conflicting reuse fails closed.
   - Ordinary records use `format_version: 1`. Compiler-published stage results use compact `format_version: 2` with scalar status plus one exact `result_artifact_binding={ref,sha256,size_bytes,body_sha256}` into the cycle packet CAS. Write the canonical result body once, then append only its compact envelope. Explicit format-v0/versionless legacy and format-v1 records remain readable; unknown versions, malformed JSONL, duplicate event IDs, cross-cycle rows, and invalid result bindings fail closed.
   - In an enforced cycle, reject public direct append. Accept only opaque compiler-owned bindings for registered producer/kind pairs: result/`compiled_stage_result_ref`, system/`compiled_system_event_ref`, observer/`compiled_stage_observation_ref`, and terminal lifecycle/`compiled_terminal_lifecycle_ref`; never accept a caller-authored full event dictionary or producer string as provenance. Re-derive system semantics and revalidate cycle-local result CAS/projection bindings at ledger consumption.
   - Treat unsealed protocol-v1 and unmarked or unsealed protocol-v2 cycles as historical read-only debt. Only an explicitly initialized, provenance-sealed protocol-v1 cycle retains compatibility writes. Missing protocol metadata is invalid/read-only, never implicit v1.
   - Require the cycle mutation contract before an effectful renderer or dashboard prewrite. The generic packet API is diagnostics only for the exact bound, initialized, provenance-sealed protocol-v1 cycle; do not expose an unbound public packet call.
   - Keep raw and hydrated readers explicit. Raw inspection returns the stored compact row; decision consumers reopen the bounded CAS artifact, verify path containment, size, raw digest, canonical-body digest, and compatibility echoes, then hydrate it. A missing or corrupt result body is a ledger error, never an empty successful stage. Readers share an existing lock but never create a cycle directory or lock; lockless legacy reads require a stable pre/post snapshot and fail closed on a racing writer.
   - Treat `status` as the canonical workflow-stage lifecycle status. Preserve owning subskill statuses such as run `success` in the subskill result artifact or `source_status`/`result_status`, not as the ledger stage status.
   - When using `python3 -m orchestrate_task_cycle ledger`, rely on its normalization of incoming stage `success` or `succeeded` to `complete`.
   - Raw subskill-result append is protocol-v1 compatibility only. For v2, use `stage prepare`, exact compiled owner/semantic/routing bindings with `stage submit`, and `stage advance`; the compiler owns the final event envelope.
   - Preserve task-pack routing fields when present: `selected_task_source`, `task_pack_id`, `task_pack_path`, `task_pack_status`, `task_pack_item_id`, `promoted_item_id`, and `completed_item_id`.
   - Preserve evidence-lifecycle and Part J contract fields when present: `instrumentation_exercise_gate`, `instrumentation_first_fire_gate`, `acceptance_encoding_gate`, `acceptance_scenario_gate`, `verifier_surface_hardening_gate`, `run_disposition`, `candidate_degraded`, `runtime_config_echo`, `config_overrides`, `command_argv`, `command_provenance_missing`, `blocker_actionability_gate`, `blocker_opacity`, `stochastic_feasibility_gate`, `execution_starvation`, and `recent_cycle_run_id_count`.
   - Preserve Part K contract fields when present: `expectation_anchor`, `designated_baseline`, `expectation_anchor_missing`, `expectation_lineage_stale`, `parity_axes`, `parity_axis_status`, `parity_unverified`, `adoption_axis_classification`, `required_output_classes`, `majority_vote_adoption`, `provisional_adoption`, `measured_but_disqualified`, `required_evidence_resolution`, `observed_evidence_resolution`, `resolution_downgrade`, `surrogate_resolution_basis`, `report_key_divergence`, and duplicate report-key path/value evidence.
   - Preserve Part L contract fields when present: `production_lane_identity`, `current_decision_lane`, `lane_identity_missing`, `pass_on_stale_lane`, `required_new_run_id`, `stale_measurement_artifact`, `decision_metadata_revision`, `axis_starved_by_missing_producer`, `producer_supply_required`, `portfolio_quota_exceeded`, `portfolio_quota_mode`, `unreachable_within_cycle`, `long_run_launch_required`, `metric_basis_inputs`, `basis_overclaim`, `actual_basis_class`, `field_class_map_missing`, and `surface_field_defect_matrix`. For cycle-unreachable long runs, transport the full `cycle_reachability_gate`, `residual_acceptance`, `harvest_validation_plan`, and any harvest receipt/recomputed gate without reducing them to booleans; pending-run projections must retain their gate digest, run ID, and harvest-plan ID.
   - Preserve Part O/O5 retention fields when present: `record_retention_policy`, `retention_class`, `rotation_manifest`, `archive_ref`, `retention_exclusion`, and `retention_fail_quiet`.
   - Preserve S10 blocker-persistence fields when present: `blocker_signature`, `first_seen_generation`, `consecutive_generation_count`, `chronic_threshold`, `chronic_threshold_unverified`, and `chronic_blocker`. These fields are debt visibility and derive-ranking input only; do not convert them into stage status failure.
   - Preserve S13 terminal-outcome latch fields when present: `terminal_justified`, `terminal_outcome_key`, `terminal_outcome_family_key`, `terminal_latch_key_version`, `blocker_signature`, `input_state_fingerprint`, `authority_state_fingerprint`, `external_state_fingerprint`, `terminal_outcome_key_unverified`, `input_delta`, `material_delta`, `quiescent_terminal_latched`, `required_missing_input_count`, and opaque required input or authority ids. These fields are terminal wait-state hygiene only; do not convert them into validation or progress verdicts.
   - Preserve S14 hook-demand fields when present: `adapter_hook_demand`, `decision_relevant_skip`, `hook_demand_threshold`, `hook_demand_threshold_unverified`, `hook_supply_required`, `demanded_hooks`, `hook_demand_unaccounted`, `hook_registry_status`, `hook_registry_unverified`, and `defined_hooks`. These fields are demand-ledger/select-routing evidence only; do not convert them into ledger-stage failure.
4. Save generated subskill packets under `packets/*.md` or `packets/*.json` and link them from the relevant event.
5. When an event links an artifact whose path and hash are identical to a previous ledger artifact, use `artifact_refs[].unchanged_ref: {path, sha256}` and `unchanged_refs` instead of reserializing the same packet content in the ledger body. The deterministic writer computes this automatically for existing files.
   - Treat hashes already stored in prior `artifact_refs` as immutable event-time evidence. Never re-hash a historical path to decide whether a new artifact is unchanged.
   - For live-run command provenance, record the full redacted `command_argv` in the run packet once. Later ledger events should point to the packet path/hash with `unchanged_ref` rather than duplicating the argv body.
6. Finalize only an identity-bound completion result with `final_candidate: true` and all six existing verdict axes. Use `python3 -m orchestrate_task_cycle ledger finalize`; do not append registry/root/seal/current completion state independently.
   - Keep the logical `attempt_id` stable across label or task-family corrections. Require the candidate to echo the complete expected-previous revision/attempt/token tuple, including an all-null first publication. Under the cycle lock, let the helper compare that tuple with current state and assign `attempt_revision` plus same-attempt supersession fields.
   - Publish `.task/cycle/<cycle-id>/finalizations/<finalization-token>.json` as the immutable content-bound snapshot, then compare-and-swap `.task/cycle/<cycle-id>/current_finalization.json` as the only authoritative visibility point. A failure before the pointer replacement may leave an unreferenced immutable snapshot, but the previous current truth must remain unchanged.
   - Bind idempotency to the decision fields only: cycle/attempt identity, expected-previous tuple, six verdict axes, and durable-state candidate. A timestamp, report label, task name, or family-name trace outside that material must not create another revision. A corrected label that changes an accepted durable projection remains a same-attempt correction and receives supersession lineage.
   - For `typed_operations`, accept only the exact producer contract `contract_version: 2`. Require an opaque `producer`, the outer `attempt_identity`, ordered `operation_ids`, ordered `expected_target_revision_ids`, `operation_set_digest`, and `candidate_sha256`. Every operation must carry deterministic content-bound `operation_id` and `idempotency_key`, `target_kind`, opaque `target_ref`, registered `operation_kind`, `expected_revision_id`, the same `attempt_identity`, ordered dependencies, `payload_schema_id`, complete replayable `payload`, `payload_digest`, and `recovery_policy_id`; compatibility aliases, when emitted, must exactly echo the canonical fields. Reject anonymous or v1 typed candidates instead of silently upgrading them.
   - Permit an empty operation set only through the exact `no_durable_state_change` receipt built by the ledger helper. It must bind the same attempt, a bounded reason ID, an empty projection, and the canonical no-change evidence digest. A generic empty `complete_projection`, missing reason, or any recurrence/debt/restriction change is invalid.
   - Reject source/work/artifact paths, locators, quotations, titles, offsets, exact character counts, raw text, and path-like strings inside durable projection payloads before writing a snapshot. Permit a generic workflow `target_ref` only outside the payload of a typed operation; it is routing metadata, not stored domain truth.
   - Compare every operation's expected target revision with the verified predecessor projection before preparing writes. For `no_durable_state_change`, compare every target observation against the same predecessor's exact registered target key: a missing key is absent, while a present key must revalidate under that target owner's schema and match its `resulting_revision_id` and `payload_digest`. Never borrow a sibling target's state or trust a caller's absent claim over a live owner projection. Any mismatch follows the pending-conflict path before snapshot or pointer publication. Reload and hash the immutable snapshot before switching the authoritative pointer, then reload the published state and verify predecessor projection, operation set, post-write target revisions, and aggregate projection digest. Require `state_commit_status: committed`, a closed `authoritative_final: success|failure|blocked|partial|not_evaluated` verdict, final-candidate and validation-axes digests, authoritative projection ID/digest, snapshot ref/hash, token, and receipt hash. An exact retry is idempotent; different content with the same idempotency key, a stale target revision, or a stale predecessor fails closed.
   - A dirty or revision-conflicted target remains `pending_conflict` and is not authoritative current. Merge or retirement must bind the pending-record hash, evidence ID/ref/digest, bounded rationale, resolution kind, committed finalization token, and canonical resolution-record digest. Only a loader-verified `merged` or `retired` resolution may unblock ordinary successor selection.
   - Run `python3 -m orchestrate_task_cycle ledger verify-finalization` before issue, derive, dashboard, report, closeout, or next-cycle consumption. Verification must bind the receipt to the current pointer and immutable snapshot, not merely validate JSON shape.
7. Render `dashboard.md` in Korean and `final_report.md` from ledger/stage evidence near the end of the cycle. Project validation/progress only from the re-verified current finalization snapshot; do not choose a conflicting latest stage boolean. Treat dashboard/profile files as snapshots with an `event_count`; if closeout appends another ledger event after rendering, do not claim those files are post-closeout snapshots unless they were regenerated after that event.
8. During close hygiene, apply `record_retention_policy` only when supplied by the adapter or caller packet. Hot records stay original, warm/cold large non-consumed packet bodies may be replaced by stubs plus archive references, and immutable evidence remains original regardless of class: loop ledgers, progress registries, validation scalars, visible-delta records, and disposition provenance. If the hook is absent, fail quiet and rotate nothing.
9. After terminal has already been justified by loopback S12 or equivalent caller evidence, compare the v2 tuple `(terminal_outcome_family_key, blocker_signature, input_state_fingerprint, authority_state_fingerprint, external_state_fingerprint)` with the previous terminal wait record. Continue reading stored v1 three-field latches only against another explicit or inferred v1 record; do not reinterpret an unversioned v1 row as v2 merely because it carried `blocker_signature` as ordinary stage metadata. If the applicable tuple is identical, emit `quiescent_terminal_latched=true`, suppress a new full cycle/snapshot, and durably append a compact `terminal_latch_observation` containing the prior reference and SHA-256, latch key, incremented streak, observation timestamp, and `material_delta=false` to the prior cycle ledger. Treat a repeated `material_delta=true` with an unchanged tuple as stale, not as another reopen. Timestamp changes, task renames, identical artifact references, and report reserialization are not material deltas. Do not create an external latch while any residual is self-resolvable or unverified. Before creating a reopened cycle, require and durably record one atomic `lifecycle_transition_result` that updates seal, registry, pack, and index; block an incomplete receipt.
10. Accumulate S14 `adapter_hook_demand[hook].decision_relevant_skip_count` across generations when packets report decision-relevant `*_unverified` fail-quiet skips. If the count reaches `hook_demand_threshold`, record `hook_supply_required=true` and `demanded_hooks`; the next derive may count only callable hook wiring, hook-free observability proof, explicit descope, terminal blocker, or escalation as `goal_productive` for that missing-hook path. Missing or malformed `hook_demand_threshold` emits `hook_demand_threshold_unverified` and keeps the existing warn-only behavior. A demanded hook is wired only when a callable adapter function exists; task-pack strings, metric names, scalar labels, or claimed hook ids are not wiring. Missing callable-hook evidence emits `hook_registry_unverified` when wiring verification was otherwise requested.

## Guardrails

- Use the helper's per-cycle lock, fsync-backed JSONL append, typed internal producer, and atomic replacement for derived snapshots. Do not bypass it with public/ad hoc concurrent writes; a malformed or unsupported ledger must be repaired explicitly rather than skipped.
- Do not mark a stage `complete` unless the owning skill produced the evidence.
- Do not loosen transition validators to accept raw subskill result vocabulary as lifecycle status; normalize at the ledger writer boundary.
- Do not append a missing, empty, or noncanonical `step`; noncanonical steps require explicit malformed-event intent and must not appear as normal dashboard stages.
- Do not rely on `$validate-subskill-result-contract` alone to supply a missing `step`; either fix the event JSON or overlay `--step` at append time.
- Do not treat a `running` execution as `success`; record it as `running` with monitor and stop evidence.
- Do not omit full redacted `command_argv` for live execution events when the run packet supplies it. If argv is missing, preserve `command_provenance_missing` so downstream consumers do not use the event as reproducible baseline evidence.
- Do not collapse `instrumentation_first_fire` into ordinary success or goal progress. Record the first-fire event as its own evidence credit when supplied.
- Do not drop Part K lineage fields during event append/rendering. Ledger snapshots may reference the packet by path/hash, but downstream consumers must still be able to recover stale expectation, parity, adoption, resolution, and report-key divergence evidence.
- Do not drop Part L lane-lineage or premise-supply fields during event append/rendering. Ledger snapshots may reference the packet by path/hash, but downstream consumers must still recover stale-lane pass, stale decision measurement, producer-starved axis, restrictive quota, cycle-unreachable target, basis overclaim, and surface-field defect evidence.
- Do not rotate, summarize, or archive away Part O/O5 immutable evidence categories: loop ledgers, progress registries, validation scalars, visible-delta records, and disposition provenance remain original regardless of retention class.
- Do not treat S10 `chronic_blocker` as a ledger-stage failure or completion verdict. It is visibility debt only and must remain separate from S6 advice-consumption debt.
- Do not treat S13 `quiescent_terminal_latched` as a new terminal judgment, blocker family, validation verdict, or progress verdict. It only prevents repeated no-progress cycle spawning after terminal was already justified and `input_delta==0`.
- Do not latch a v2 record from a global terminal label or from one key alone. Family, blocker, input, authority, and external-state fingerprints are required; a reopen with only a subset of seal/registry/pack/index updates is incomplete.
- Do not treat S14 `hook_supply_required` as a ledger-stage failure, second gate, or hard stop. It only constrains next-task selection for the missing-hook path, and only callable adapter functions count as wired hooks.
- Do not overwrite existing stage history. Append a corrective event with a reason.
- Do not expose a candidate, prepared mutation, orphan finalization snapshot, or partially written projection as current. Only the verified current pointer and receipt are authoritative.
- Do not accept a receipt whose schema, attempt revision, supersession, token, snapshot hash, six-axis digest, or authoritative projection digest differs from the current immutable snapshot.
- Do not reserialize identical packet bodies across cycle events when the path/hash are unchanged. Preserve `unchanged_ref(path+hash)` so cycle-efficiency profiling can distinguish fixed-cost work from repeated artifact payloads.
- Do not copy a format-v2 result body back into `stage.jsonl`, `current_stage.json`, dashboards, or reports. Preserve its binding and hydrate only at the consumer boundary.
- Do not treat record retention as completion, progress, or blocker evidence. It is close hygiene only and must not be double-counted as Part M disposition or Part N persistence policy.
- Do not edit repository source, tests, notebooks, runtime/build configuration, or other behavior-changing files from this skill.
