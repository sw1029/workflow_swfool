# Orchestrate Task Cycle

## Overview

Use this skill to run one complete task workflow cycle in a fixed order:

1. Ensure `task.md` exists.
2. `$manage-agent-authority`
3. `route_plan` and `$build-validation-set-with-agents` `validation_set_plan` when validation assets may be needed.
4. `$task-md-agent-governance`
5. `code_structure_audit` read-only module-boundary gate for generated or changed code.
6. `$run-task-code-and-log`
7. `$review-cycle-output-quality`
8. `$audit-cycle-loopback`
9. `$build-validation-set-with-agents` `validation_set_build` when the task produces, refreshes, or consumes reusable validation assets.
10. `$manage-schema-contracts` before deriving the next task
11. `$derive-improvement-task` for the next `task.md` or a task-pack planning transaction that promotes at most one item into `task.md`
12. `$manage-schema-contracts` after deriving the next task
13. `$manage-task-state-index`
14. `$validate-task-completion`
15. `$manage-implementation-issues`
16. `$repo-change-commit` explicitly after validation when the current workspace is inside a Git worktree, including coherent `partial` validation states

The main agent coordinates the cycle and preserves the ordering. Each underlying skill still owns its specialized work, agents, logs, indexes, audits, and guardrails.

Use [workflow-routing.md](references/workflow-routing.md) as the canonical routing and transition reference for this skill. It standardizes subskill packet targets, required inputs, reasoning/model routing, `.agent_goal` GT propagation, running-state handling, and forbidden bypasses. Use [cycle-artifacts.md](references/cycle-artifacts.md) for ledger, result-contract, visible-increment, validation-scope, evidence-cache, dashboard, and efficiency-profile artifact contracts. Use [task-pack-workflow.md](references/task-pack-workflow.md) for optional `.task/task_pack` JSON queues, user-language Markdown renders, pack item promotion, insert/reorder rules, blocker signatures, positive input delta gates, and terminal blocker state. Do not create any new long-term GT document for routing; use only the existing `.agent_goal/*.md` files listed there.

## Canonical Full Phase Order

Run the full phases in this fixed order. The orchestrator remains a coordinator; implementation code changes still belong only to `$task-md-agent-governance`.

1. `context`: collect compact workspace context with separate `available_goal_truth` and `used_goal_truth`.
2. `ledger_init`: use `$maintain-cycle-ledger` to initialize `.task/cycle/<cycle-id>/`.
3. `authority`: resolve `$manage-agent-authority`.
4. `acceptance`: use `$normalize-acceptance-and-demo`.
5. `route_plan`: classify the cycle route, including whether validation assets are `not_applicable`, `plan`, `build`, `refresh`, `consume`, or `blocked_or_candidate_only`.
6. `validation_set_plan`: invoke `$build-validation-set-with-agents` in `plan` mode when evaluation assets, oracles, source-class boundaries, split policy, leakage policy, or validation-set gaps are relevant; otherwise append `not_applicable` with a reason.
7. `governance`: invoke `$task-md-agent-governance`.
8. `result_contract`: use `$validate-subskill-result-contract` on the governance result.
9. `ledger_append`: append governance/contract evidence through `$maintain-cycle-ledger`.
10. `code_structure_audit`: run the read-only generated-code structure/module-boundary gate.
11. `run`: invoke `$run-task-code-and-log`.
12. `qualitative_review`: invoke `$review-cycle-output-quality` to run exactly one read-only direct qualitative reviewer agent over task output artifacts.
13. `loopback_audit`: invoke `$audit-cycle-loopback` to compute `anti_loop_progress_gate` from raw artifacts and the family progress registry.
14. `validation_set_build`: invoke `$build-validation-set-with-agents` in `build`, `refresh`, or `consume` mode when the cycle produces, updates, or uses reusable validation assets; otherwise append `not_applicable` with a reason.
15. `schema_pre_derive`: invoke `$manage-schema-contracts`.
16. `visible_increment`: use `$record-visible-increment`.
17. `derive`: invoke `$derive-improvement-task`; `$optimize-task-slice` may provide advisory input only.
18. `schema_post_derive`: invoke `$manage-schema-contracts`.
19. `index`: invoke `$manage-task-state-index`.
20. `validate`: invoke `$validate-task-completion`.
21. `issue`: invoke `$manage-implementation-issues`.
22. `commit`: invoke `$repo-change-commit` for the implementation/workflow change set when applicable.
23. `dashboard`: use `$render-cycle-dashboard` and `$profile-cycle-efficiency`.
24. `report`: assemble the Korean cycle report.
25. `closeout_commit`: invoke `$repo-change-commit` for report/dashboard/ledger/advice closeout artifacts when applicable.

Use `$manage-external-advice` for `.agent_advice/` direction documents when present. These documents are non-GT planning evidence: include them in `used_advice`, keep them out of `used_goal_truth`, and report them under `비-GT 방향성 문서`. Active advice packets passed to subskills must include at least the advice ID/path, summary, actionable directives, normalization fidelity status, raw-direct-reference requirement, and application gates when those fields are present in the normalized advice file.

Active or adopted `.agent_advice` files used by the cycle are intentional workflow evidence and default to tracked/committed artifacts. Mark advice as `local_only` only for sensitive or explicitly external-only material, and carry the reason into the report and commit packet.

The coordinator must not directly modify repository implementation code. Source files, tests, notebooks, scripts, runtime configuration, build/CI configuration, and other behavior-changing implementation files may be edited only inside the `$task-md-agent-governance` step. Other steps may write only their owned workflow/state artifacts, such as `task.md`, `.task/`, `.agent_log/`, `.schema/`, `.contract/`, validation reports, Git metadata/staging decisions, and non-runtime `.gitignore` housekeeping.

`$build-validation-set-with-agents` owns validation asset production, not implementation or final completion judgment. It may write `.validation/sets/`, `.validation/candidates/`, `.validation/registry.jsonl`, `.validation/index.md`, and cycle-local `.task/validation_set/` evidence. It must preserve source class, avoid raw body persistence, keep `not_gold: true` unless human-reviewed or fully deterministic authoritative evidence exists, and route any validator/test/source code changes back through `$task-md-agent-governance`.

Use `$manage-agent-authority` as the owner of `.agent_goal/agent_authority.md`. Resolve the effective authority policy through that skill before task derivation, implementation delegation, validation, or final reporting. When the file is absent, ask `$manage-agent-authority` for the `default_current_agent_permissions` fallback or `ensure_default` when durable policy is needed; never infer extra authority, network/API access, destructive permissions, or expanded autonomy from the absence of the file. The authority policy may constrain API calls, external service use, allowed direction variation, strictness, implementation-first versus validation-first posture, artifact/output confirmation priority, quality-review priority, conservative implementation, and escalation behavior. It can narrow or prioritize within the current session's permissions, but it cannot override higher-priority instructions or grant capabilities the running agent does not have.

When invoking `$task-md-agent-governance`, pass the agent routing requirement explicitly: governance, audit, integration-review, miss-cleanup, and schema-contract work must run with high reasoning when the delegated tooling exposes `reasoning_effort`; ID-only consistency work is routed through `$manage-task-state-index` with fixed `reasoning_effort: medium`. Actual code-writing workers that would previously have used Spark must be spawned as workers on `model: gpt-5.5` with `reasoning_effort: medium` by default. Use `model: gpt-5.5` with `reasoning_effort: high` for high-reliability core logic, including schema/contract compatibility logic, task/index/issue lifecycle state mutation, validation gates, security-sensitive behavior, irreversible workflow decisions, or code whose regression can corrupt durable artifacts. If a tool cannot enforce those fields, encode the routing requirement in the prompt and report the limitation.

When invoking `$derive-improvement-task`, pass the late-cycle improvement routing requirement explicitly: goal/convention alignment, task_miss analysis, issue goal-fit, improvement-analysis, synthesis, candidate-prioritization, schema-contract planning, and ID-consistency agents must use fixed `reasoning_effort: xhigh` whenever the delegated tooling exposes it. If a tool cannot enforce the field, encode the xhigh requirement in the prompt and report the limitation.

When invoking `$manage-task-state-index` for ID scan, audit, link repair, lifecycle transition recording, candidate/miss cleanup gating, or optional ID insight agents, pass fixed `reasoning_effort: medium` whenever the delegated tooling exposes it. This medium ID-correction routing is specific to `$manage-task-state-index` and should be reported if tooling cannot enforce it.

When invoking code-analysis subskills such as `$inspect-repo-with-agents` or `$inspect-oom-risk`, pass `reasoning_effort: high` as the minimum for ordinary code analysis. Use `reasoning_effort: xhigh` for important work review, including final completion validation, post-implementation governance that can create or clear blockers, miss/candidate/issue cleanup decisions, schema/API/CLI/data-contract review, security-sensitive review, destructive or irreversible workflow decisions, and high-severity regression analysis. ID-only correction remains the `$manage-task-state-index` medium exception.

When invoking `$repo-change-commit` for Git status classification, staging, commit readiness, or actual commit creation, pass fixed `reasoning_effort: low` whenever the delegated tooling exposes it. This low-effort commit routing is specific to `$repo-change-commit`; do not inherit high/xhigh analysis routing into commit finalization. If tooling cannot enforce the field, encode the low-effort requirement in the prompt and report the limitation.

Long-term execution is acceptable when the user, `task.md`, or governance result explicitly authorizes it. A `$run-task-code-and-log` result of `running` is valid execution evidence for an in-progress cycle, but it is not completion evidence unless the task's success criteria are satisfied by startup/heartbeat evidence alone.

When invoking `$review-cycle-output-quality`, pass the run result, generated output artifact paths, task acceptance criteria, authority policy, no-overclaim constraints, enriched active advice packet, and relevant schema/contract summaries. Require exactly one read-only reviewer agent with `reasoning_effort: xhigh` when the delegated tooling exposes it. The reviewer may directly inspect output artifacts, generated reports, KG/candidate artifacts, logs, and validation bundles named by cycle evidence, but must not edit files or persist raw sensitive/source/provider bodies. Treat the review as agent qualitative evidence, not goal truth and not direct human approval.

When invoking `$build-validation-set-with-agents`, pass the task packet, authority policy, actually used GT, enriched active advice packet, schema/contract summaries, validation gaps from `.task/task_miss/` and `.issue/`, existing `.validation/` inventory, source-class policy, no-overclaim policy, run evidence, and qualitative review findings when available. Require deterministic scripts and executable oracles before agent consensus when possible. Require output fields for validation-set ID, status, quality tier, `not_gold`, item/label/oracle counts, oracle/split/leakage/root paths, source-class distribution, no-overclaim flags, blockers, and advice disposition.

## Goal Alignment And Portfolio Guard

Before `$derive-improvement-task`, build a compact portfolio planning packet from current task evidence, run evidence, the `$review-cycle-output-quality` result, validation-set evidence, validation evidence, active issues, `.agent_goal/` GT, `.agent_advice/` non-GT direction, schema contracts, and progress-loop detection. This packet is part of the `derive` stage, not a new ledger phase unless the local scripts explicitly support one. For repeated safe-but-stationary cycles, apply [anti-loop-progress-gates.md](references/anti-loop-progress-gates.md) as derive policy.

The packet must contain:

- `goal_delta`: final-goal axes, current-task axes, `goal_alignment_gap`, no-claim boundaries, and recommended retarget options.
- `blocker_taxonomy`: classify blockers as `provider_runtime_blocker`, `output_artifact_blocker`, `kg_core_blocker`, `claim_readiness_blocker`, `rights_blocker`, `task_state_blocker`, `quality_blocker`, or `workflow_lifecycle_blocker`.
- `portfolio_candidates`: compare at least the current blocker-following task against provider-neutral KG/core/evidence/feature/commitment/claim/rights/gold-review or task-state cleanup options when those axes are relevant to the goal.
- `provider_dependency`: mark each candidate as `none`, `optional`, `bounded`, or `required`; do not let provider-dependent work monopolize derivation when provider-neutral work can advance safely.
- `parallel_subwork`: list read-only or disjoint-write sub-work packages that can run inside one canonical active task without creating multiple active `task.md` states.
- `advice_disposition`: for each active/adopted advice file, record `not_applicable`, `deferred`, `partially_applied_as_guard`, `accepted_as_design_candidate`, `incorporated_into_task`, `implemented_in_schema`, `rejected`, or `retired_or_superseded`.
- `qualitative_review_packet`: include `review_status`, `quality_verdict`, reviewed artifacts, key findings, blocker taxonomy deltas, direction recommendations, semantic readiness/cap fields, no-overclaim flags, and evidence paths from `$review-cycle-output-quality`.
- `validation_set_packet`: include `validation_set_need`, `validation_set_status`, `quality_tier`, `not_gold`, `validation_set_id`, item/label/oracle counts, source-class distribution, oracle/split/leakage/root paths, sealed holdout status, and blocked/candidate-only reasons from `$build-validation-set-with-agents`.
- `output_delta_packet`: include output-delta contract path/status, repository-declared output layer paths, previous/current fingerprints, `changed_vs_previous`, `semantic_progress`, `produced_domain_delta`, `metadata_only`, `effective_progress_kind`, scalar counts, evidence paths, and skipped/blocked reason from `scripts/output_delta_contract.py` or `$review-cycle-output-quality`.
- `anti_loop_progress_gate`: include `$audit-cycle-loopback` output: family key, suffix-normalized `root_key`, `root_family_key`, `blocker_root_family`, artifact family, semantic signature, `changed_vs_previous`, `semantic_progress`, `authoritative_semantic_progress` when present, same-family micro-hardening count, `validator_integrity_gate`, `previous_accepted_baseline`, `coverage_quality_delta_gate`, `coverage_quality_delta_reconciliation_gate`, `substance_delta_gate`, `vacuous_corrective_gate`, `facet_root_map_applied`, `advice_freshness_gate`, `structure_metrics_gate`, `provider_scale_dispatch_gate`, `measurement_progress`, `measurement_progress_allowed`, `measurement_streak`, `measurement_progress_streak_for_root_key`, `measurement_progress_streak_for_root_family`, `task_correction_class`, `detection_only_streak_for_root_family`, `requires_correction_or_terminal`, `blocker_signature`, `blocker_ladder_rung`, `blocker_mutation_kind`, `forward_mutation_budget_remaining`, `force_implementation_cycle`, `effective_allowed_dispositions`, `disposition_intersection_basis`, `consolidation_streak`, provider/env behavior booleans, quality vector, high-water marks, recommended disposition, hard-stop state, findings, and evidence paths.
- `failure_autopsy_packet`: include safe scalar failure diagnostics from `$run-task-code-and-log`, such as `error_type`, `exception_class`, `traceback_last_frame`, `http_status`, `missing_env_key_names`, `provider_request_count`, `provider_status`, `failure_class`, `provider_response_empty`, and `provider_response_parse_failed` when execution failed without exposing raw bodies or secrets.
- `code_structure_audit_packet`: include the post-governance read-only structure gate result: changed files scanned, oversize files, responsibility clusters, moduleization requirement, suggested module root, responsibility split plan, compatibility constraints, validation-scope delta, existing-debt exemptions, and evidence paths.
- `task_pack_packet`: include active pack ID/path/status, Markdown render path, current item ID/order/status, planned item count, pack-level goal, mutation log summary, terminal blocker state, `pack_disposition`, `pack_mutation_plan`, and whether the next task should create a bounded pack, promote from the pack, insert/reorder/skip/supersede pack items, derive as standalone with rationale, or record terminal state.
- `gt_constraint_conflict_packet`: include `detect_gt_constraint_conflict.py` output when task constraints or behavior booleans may forbid an action that `.agent_goal/agent_authority.md` or `.agent_goal/conventions.md` allows or requires. Include scalar behavior evidence such as `provider_request_count`, `env_file_read`, retry/probe evidence, and terminal legitimacy. If it reports `status: block`, derivation may only select a conflict-resolution task, goal-productive work that removes the contradiction, or user escalation/terminal state.
- `loop_breaker_packet`: include normalized `blocker_signature`, additive `semantic_signature`, suffix-normalized `root_key`, `blocker_root_family`/root-family counts, stable `root_axis` counts, same-root-key/signature/semantic-family counts across recent cycles, compared cycle IDs, `validator_integrity_gate`, `detection_balance_gate`, positive input delta gate status, newly introduced input kinds, `has_supplied_input_delta`, `supplied_input_artifact_paths`, provider re-attempt and mitigation gate status, command-surface budget status, sealed semantic-family matches, `autonomous_retarget_disabled`, and terminal-blocker recommendation when no viable candidate remains.
- `feature_symbol_gate`: include observed-over-declared workflow feature symbols from `scripts/detect_progress_loop.py`: `consumed_input_fp`, `target_unit_fp`, `blocker_root_axis`, `observed_output_class`, no-delta occurrence counts, registry path, terminal-history matches, and allowed dispositions. Only observed `node_edge_delta` resets the loop counter; self-declared `produced_domain_delta` never resets it by itself.
- `goal_distance_gate`: include `cycles_since_goal_productive_output`, `goal_productive_this_cycle`, `governance_only_streak`, and whether the next task must be `goal_productive` or `terminal_blocked`.
- `anti_loop_progress_gate`: include repeated artifact family, same-family micro-hardening count, provider/env behavior booleans, semantic-output stagnation, qualitative review progress cap, command-surface hard-stop state, and allowed next dispositions from `$audit-cycle-loopback` and [anti-loop-progress-gates.md](references/anti-loop-progress-gates.md).
- `advice_coherence_packet`: include root steering docs found at the workspace root, whether each is represented in `.agent_advice/active`, and any `orphan_advice_not_intaken` warning.

Use this guard to prevent repeated safe-but-stationary cycles. If recent tasks repeatedly classify provider/runtime/output absence without producing reviewable output, the next task should either finish the required handoff quickly or retarget to provider-neutral progress such as validation-set/oracle/leakage/source-class construction, KG core/evidence/feature schema, deterministic commitment/claim evaluation, rights policy graph, gold-review protocol, or task-state cleanup. Keep no-overclaim rules intact: these retarget options are design or implementation candidates, not proof of KG usefulness, ZKP readiness, rights readiness, gold quality, or final readiness.

When a single cycle needs multiple independent tracks, keep one canonical active task and express parallelism as sub-work packages owned by the relevant subskills. Parallel sub-work may gather read-only audits, design schema drafts, test plans, or fixture-based prototypes. Serialize writes to `task.md`, `.task/index.*`, `.schema/index.md`, `.schema/contracts.jsonl`, `.issue/`, and commit staging through the owning skill and integration step. Do not create multiple active tasks to simulate parallelism.

When long-range ordering is useful, keep the one-active-task rule and express the ordered work as an optional task pack under `.task/task_pack/`. A pack item becomes executable only when `$derive-improvement-task` promotes it into `task.md`; the rest of the pack remains planning state. The late-cycle derive step may create a bounded pack, promote one item, insert/reorder/skip/supersede items, or terminal-block the pack only with new evidence, repeated-blocker evidence, missing supplied positive input delta, provider-neutral retarget evidence, dependency repair evidence, or a terminal blocker decision as defined in [task-pack-workflow.md](references/task-pack-workflow.md). Treat user-requested "exclude" as an item status mutation such as `skipped` or `superseded`, not as deletion, unless a later ID audit and owning workflow explicitly authorize deletion.

## Efficiency Rules

Keep the existing workflow order, ownership boundaries, and validation gates, but avoid repeated micro-cycles that add cost without moving the blocker state.

- Prefer blocker-state-transition tasks over micro-contract tasks. A new task should normally move the repository from one meaningful state to another, such as `missing evidence -> intake path proven`, `intake path proven -> bounded preflight`, or `bounded preflight -> source-backed validation`. A task that only adds another no-live scalar/check contract is acceptable only when it unlocks a new execution state, prevents a specific unsafe transition, or is merged into a bounded batch with adjacent contracts.
- When consecutive no-live contracts share the same target script/module, issue blocker, prerequisite chain, promotion boundary, and validation surface, ask `$derive-improvement-task` to batch them into one task instead of creating sequential single-contract tasks.
- Use prerequisite validation profiles instead of full historical reruns by default. Prefer `current_only` for the current validator plus direct predecessor hash/status checks, `affected_chain` when changed surfaces can affect selected predecessors, and `full_chain` only before live dispatch, readiness promotion, issue closure, high-risk shared validator/runtime changes, or explicit user request.
- Require called skills to distinguish `validation_verdict` from `progress_verdict`. A task may be validation-passed but `progress_verdict: safety_only` when it proves fail-closed/no-live behavior without supplying source-backed evidence, producing dataset/KG artifacts, improving qualitative output, or advancing readiness.
- Require called skills to distinguish self-reported `progress_kind` from output-delta-derived `effective_progress_kind`. A task that writes only metadata, sidecars, workflow state, schema records, or measured absences is `governance_only` when the repository output-delta contract reports `produced_domain_delta: false`. A non-empty output is not enough: goal-productive output requires `changed_vs_previous=true` and `semantic_progress=true` on repository-declared primary output, or independent validated positive evidence.
- If the last two completed tasks are `safety_only` and the same issue blocker remains, the next task must either supply/validate the missing evidence path, perform a bounded source-backed run/preflight, consolidate remaining no-live boundaries into one batch task, or stop with a concrete blocker explaining what external input or authorization is missing.
- If the last two progress-bearing cycles are metadata-only after output-delta review, the next task must resume primary output, perform bounded source-backed run/preflight, repair the safe failure root cause, or record terminal state. Do not select another measurement-only cohort or metadata reconciliation as progress.
- If `anti_loop_progress_gate.measurement_progress=true`, treat it as instrumentation evidence unless `measurement_progress_allowed=true`, `coverage_quality_delta_gate.quality_delta_pass=true`, and `substance_delta_gate.substance_delta_pass=true`. Stop using the exemption after root-key/root-family cap. Do not let measurement-only work re-add `goal_productive`.
- If `anti_loop_progress_gate.blocker_mutation_kind=forward_mutation`, require derive to treat the rung movement as changed blocker state only when G-SUBSTANCE or strict changed-and-semantic output-delta evidence supports it. If `force_implementation_cycle=true`, the next task must be in-place implementation work, terminal-block with an authority/input blocker, or user escalation.
- If qualitative review reports `semantic_ready=false`, `placeholder_event_found=true`, or `surface_entity_suspected=true` for the primary artifact, cap `effective_progress_kind` at `governance_only` unless strict changed-and-semantic output-delta evidence overrides the cap.
- If recent cycles share the same normalized `blocker_signature`, require `$derive-improvement-task` to consume the next safe task-pack item, insert/reorder the active pack, supply a real input artifact/domain delta, or record `terminal_blocked`. Repeated narrowing or handoff tasks without a supplied input delta are not progress.
- Compare repeated blockers by suffix-normalized `root_key` and `semantic_signature` before raw `blocker_signature` when `target_surface`, run directory, timestamps, sequential `vN`, date, or `after-*` names vary. Preserve `blocker_signature` for compatibility, but let the root key and semantic family drive loop-breaking decisions.
- If a `semantic_signature` has been terminal-blocked or appears in a sealed family list, do not derive another non-terminal task in that family unless a supplied input delta, authority change, or external-state change is present.
- If the next task is evidence-family work and `positive_input_delta_required` is true, require a supplied positive input delta before treating the task as progress: at least one non-empty artifact path named in `supplied_input_artifact_paths` or `produced_domain_delta=true`. A newly introduced input kind, new wording, a new timestamp, or a restated blocker is only a hint, not progress.
- If `cycles_since_goal_productive_output` exceeds the local threshold, the next task must either produce goal-productive output, quality evidence, source-backed validation, or explicit terminal state. More sidecar/governance reconciliation is not enough.
- If `detect_gt_constraint_conflict.py` reports a GT/task contradiction, do not derive another task that inherits the contradictory prohibition. Select a task that resolves the task-level contradiction, select goal-productive work that removes the blocker without violating GT, or escalate to the user when no authorized task exists.
- If `root_axis_gate.autonomous_retarget_disabled=true`, autonomous provider-neutral/governance retargeting is exhausted for that root axis or suffix-normalized root key. The next derive result must be goal-productive or terminal/user-escalated; another governance-only retarget is invalid even when the semantic signature, version suffix, or task family name changes.
- Classify selected next-task progress as `progress_kind: goal_productive` or `progress_kind: governance_only`. If the last two progress-bearing cycles were `governance_only`, the next derivation must select `goal_productive` work or record `terminal_blocked`.
- If active task selection reaches zero viable candidates or pack items, record a terminal blocker with required handoff/input/authority details instead of deriving another narrowing task.
- If `failure_autopsy_packet` or `provider_reattempt_gate` shows `failure_class` in `empty|parse|rate_limit|timeout|transient|incomplete` with unexhausted mitigations, derive must not treat the provider as terminal. Required mitigation evidence is scalar-only: `structured_output`, `window_reduce`, `timeout_budget_increase`, `backoff_retry>=3`, and `model_fallback`, with unavailable/unauthorized mitigations named explicitly. When authority permits retry/probe, select bounded retry/probe and record `provider_reattempt_disposition`; otherwise terminal/user escalation must name the authority blocker instead of sealing provider failure as permanent.
- Before sealing or terminal-blocking a blocker family, require evidence that one root-cause/autopsy repair task has already been attempted for that family, or record why root-cause repair is impossible, unauthorized, or unsafe.
- Before sealing or terminal-blocking a family, require `authorized_alternative_path_exists=false` or `authorized_alternative_path_attempted=true`. If authority still allows an unattempted provider retry/probe, source-backed run, or deterministic provider-neutral quality recovery path, reject the seal and retarget.
- When terminal state is recorded, include `semantic_signature` and persist or update `.task/sealed_blocker_families.json` when the owning derive/task-pack workflow can do so. This file is workflow state, not GT, and exists only to prevent same-family non-terminal derivation without new input.
- When `detect_progress_loop.py --root . --strict` reports `status: block`, validate `$derive-improvement-task` output with `result_contract.py --target derive --mode block`. The derive result may only select `progress_kind: goal_productive` or record `selected_task_source: terminal_blocked` with provider-track and provider-neutral/quality-track attempt evidence.
- When `command_surface_budget.consolidation_candidate_required=true` or `hard_gate=true`, require derive to register or select a consolidation/refactor candidate unless strict changed-and-semantic primary-output evidence already justifies a goal-productive disposition, or the derive result records terminal/user-escalation state. Do not allow another version-suffixed `-vNNN` contract/gate/preflight/locator/recovery command on the same over-budget surface as the next task.
- When `effective_allowed_dispositions` is present, require derive and result-contract validation to enforce it as the intersection of active gate dispositions. Do not let a disposition through merely because one individual gate allowed it.
- If provider/output blockers repeat while provider-neutral validation-set, oracle, leakage-control, KG, evidence, feature, commitment, claim, rights, quality, or task-state work can advance without raw body or live-provider access, require `$derive-improvement-task` to include at least one provider-neutral candidate in the selected-task rationale. Provider work may remain an optional producer track, but it should not remain the only path to goal progress unless authority, task scope, or missing prerequisites make all provider-neutral tracks unsafe.
- Require validation and final reporting to distinguish axis-level progress from overall progress. The orchestrator, `$build-validation-set-with-agents`, `$run-task-code-and-log`, `$review-cycle-output-quality`, `$manage-schema-contracts`, `$derive-improvement-task`, `$manage-task-state-index`, `$validate-task-completion`, and `$repo-change-commit` must not directly patch code to satisfy the current task.
- Treat `task.md`, `.task/`, `.agent_log/`, `.schema/`, `.contract/`, validation reports, index/log artifacts, and non-runtime `.gitignore` housekeeping as workflow/state artifacts, not implementation-code escape hatches. Do not place source-code fixes in those artifacts.

- When active `.agent_advice/active/` exists, do not advance a validation-set plan/build result that omits both `used_advice` and an explicit advice defer/reject/not-applicable rationale.

- Do not claim validation-set gold quality unless the result carries human-reviewed evidence or fully deterministic authoritative oracle evidence. Agent consensus alone is not gold.
- Do not skip `$manage-task-state-index` before final validation when any task-state artifact exists.
- Do not claim final completion before `$validate-task-completion` returns a verdict.
- Do not call `$repo-change-commit` before `$validate-task-completion` has returned a verdict and `$manage-implementation-issues` has tracked the new task or blockers when applicable.
- Do call `$repo-change-commit` explicitly after `$validate-task-completion` when `git rev-parse --is-inside-work-tree` returns `true`; do not omit it merely because validation, issue tracking, or a raw Git status check already ran.
- Do call the closeout `$repo-change-commit` after report/dashboard rendering when closeout artifacts are intentional and the workspace is a Git worktree.

- Do not let the next-task derivation delete candidates or miss reports unless the owning skill and ID audit allow it.
- Route ID correction through `$manage-task-state-index` with fixed `reasoning_effort: medium`; do not let other routing defaults silently override this for scan/audit/link repair.

## Failure Handling

- If initial task derivation fails, stop before implementation and report why `task.md` could not be created.
- If any non-governance step discovers that code must change, do not edit it in place. Record the gap, keep or update `task.md` as appropriate, and rerun or schedule `$task-md-agent-governance`.
- If implementation or audit produces blocking task_miss items, still index them and let validation classify the cycle.
- If execution fails, log the failure through `$run-task-code-and-log`, index it, then validate as `partial` or `failed`.
- If execution is still `running`, log and index the active run, preserve monitor/stop details, defer final-output-dependent task derivation or completion claims, and validate as `partial` unless the task's success criteria are already satisfied.
- If qualitative output review fails or is blocked, preserve run/log/index state, record whether the blocker is missing artifact persistence, forbidden content access, unavailable reviewer delegation, or unreviewable output shape, and use that finding as direction evidence for derivation or validation rather than editing code directly.
- If validation-set planning or building fails or is blocked, preserve run/log/review/index state, record whether the blocker is missing source evidence, source-class ambiguity, raw-body authority, oracle infeasibility, leakage, label disagreement, or holdout sealing, and use that finding as direction evidence for derivation or validation rather than editing code directly.
- If deriving the next task fails after implementation, preserve logs and index state, then validate the completed task with a handoff gap.
- If schema contract refresh or reconciliation fails, preserve task/log/index state and report schema contract management as a separate blocker or handoff gap.
- If issue tracking fails after validation, preserve validation and index state, report issue tracking as a separate blocker or handoff gap, and continue to Git status inspection only when doing so is safe.
- If ID audit finds high-severity issues, do not delete artifacts and do not report `complete` unless `$validate-task-completion` explicitly resolves or downgrades them.
- If Git finalization fails after validation, preserve the validation verdict and report Git finalization as a separate failure or blocker.
