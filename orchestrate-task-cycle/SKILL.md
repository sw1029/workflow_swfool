# Orchestrate Task Cycle

## Overview

Use this skill to run one governed repository task cycle from task discovery through implementation routing, execution, review, loopback, current-task validation and issue handling, next-task derivation, Git finalization, dashboard/report rendering, and closeout.

Use `authority.operations.json` with [authority-boundary-contract.md](references/authority-boundary-contract.md) and the shared [authority v2 contract](../manage-agent-authority/references/authority-v2-contract.md). Inspection is authority-free; worker dispatch, external dispatch, and task-topology mutation use distinct exact operations and may not share or infer a grant. A topology effect first uses the ordinary `mutate_task_topology` grant. Its post-effect activation may use `activate_task_topology_settlement` only as a bound-lifecycle operation after the exact reservation, pre-commit verification, execution-result receipt, and owner consume receipt all validate; it is not a second authority source.

The main agent is the coordinator. Each called skill owns its own implementation, validation, logging, indexing, schema, issue, and commit behavior. Keep the orchestrator summary-first: pass compact packets, IDs, paths, verdicts, blockers, and changed-file lists rather than loading implementation bodies into the main session.

## Domain Adapter Contract

The orchestrator normally receives adapter-derived fields through subskill packets. When close accounting has direct adapter access, it may consume these optional Part O/P hooks without interpreting domain-specific meanings:

- `goal_axis_map(targets=None, quality_vector=None, **context) -> list|dict`: O2 helper returning adapter-owned goal axis ids, delta basis fields, and optional `axis_delta`, `axis_stall_streak`, direct-advancement task class, or terminal-blocker evidence. The adapter owns axis definitions, delta semantics, and thresholds. If absent or malformed, fail quiet and keep existing close, dashboard, and derive routing unchanged.
- `landed_feature_inventory(**context) -> dict`: P1 helper returning adapter-owned landed feature inventory and authorized opaque feature classes for representative outputs. If absent or malformed, fail quiet and keep existing review/loopback routing unchanged.
- `producer_execution_evidence(run_id, **context) -> dict`: P2 helper returning whether a run performed fresh producer execution plus opaque evidence fields. If absent or malformed, fail quiet unless normalized acceptance requires freshness verification.
- `input_source_lineage(run_id, **context) -> dict`: P4 helper returning opaque input-source lineage and generation-time defect dominance. If absent or malformed, fail quiet and keep existing loopback/derive routing unchanged.
- `governance_packet_budget(**context) -> dict`: P5 helper returning adapter-owned governance packet budget scalars. If absent or malformed, fail quiet and keep existing terminal packet recording unchanged.
- `disposition_applicability(blocker, constraint_state, **context) -> dict`: S1 helper returning whether a candidate disposition actually moves the current adapter-owned constraint plus an opaque reason and optional residual blocker class. Use it only to filter disposition options before presenting `effective_allowed_dispositions`, terminal blocker options, or user escalation options. If absent or malformed, fail quiet, preserve the existing disposition options, and carry `applicability_unverified` warning evidence.
- `self_resolvable_input(required_input, **context) -> dict`: S4 helper returning `resolvable: bool`, `kind`, and optional opaque evidence ids for a missing input that can be generated by deterministic rewiring of existing repository capability without new authority. Prefer this hook when supplied; otherwise consume Q1 `self_resolvable_input_probe`. If absent or malformed, fail quiet, preserve existing positive-input-delta terminal/escalation routing, and carry `self_resolvable_input_unverified` when escalation remains.
- `self_resolvable_input_probe(required_missing_input, **context) -> dict`: Q1 helper returning whether a missing input can be generated by deterministic rewiring of existing repository capabilities without new authority. If absent or malformed, fail quiet and keep existing positive-input-delta terminal/escalation routing unchanged.
- `feature_presence_evidence(feature_class, artifact_root, **context) -> dict`: Q2 helper returning body-field presence evidence for landed feature inheritance. The reviewer owns consumption; the orchestrator preserves only scalar fields such as `artifact_body_missing`, `presence_basis`, and `synthesis_not_materialized`.
- `hook_registry(**context) -> dict`: Q3/Q5 helper returning `defined_hooks` for provenance validation and hook-debt accounting. If absent or malformed, fail quiet except schema-contract debt may record the missing registry as warn-only visibility evidence.
- `reason_code_rank(reason_codes, **context) -> dict`: Q4 helper returning `primary`, `secondary`, and `precedence_edges` for fail-close reason codes. If absent or malformed, fail quiet and keep existing derive ordering unchanged.
- `code_convention_contract(**context) -> dict`: existing Q6 contract source for code-structure audit, including purpose directory taxonomy, role thresholds, allowed depth, reuse or abstract roots, and dependency-layer order. If absent or malformed, depth/fan-out findings are warn-only and the absence is visible only through Q5 hook debt when a consumer attempted to use the contract.

Do not hardcode goal axes, thresholds, proxy names, feature classes, producer-execution meanings, input-generation meanings, packet budgets, reason-code meanings, hook registries, body-field meanings, code-depth taxonomies, reuse or inheritance roots, dependency-layer order, or direct-advancement meanings in this skill. Part O/O2' is aggregate close accounting only; do not double-count it as B/G-series individual-cycle dilution or N1' vacuity. Part P fields are in-place revisions of existing review, acceptance, loopback, derive, validation, and terminal recording decisions; Part Q fields revise existing routing, review, provenance, derive ranking, hook-debt visibility, and code-structure audit decisions. They do not create a new phase, detector, report, or skill.

## Core Invariants

- Keep pre-validation evaluation write-free. Run, qualitative review, and loopback emit immutable observations or prepared state mutations; they must not update a current registry, root-cause ledger, blocker seal, or active attempt pointer. Completion validation emits an identity-bound `final_candidate` with the existing six verdict axes and may only preserve or downgrade earlier claims. Finalize that candidate inside the existing `validate` boundary with `python3 -m orchestrate_task_cycle ledger`: assign the same-attempt revision/supersession relation, publish one immutable snapshot, atomically compare-and-swap the current pointer, and issue a content-bound `cycle_finalization_receipt`. Only a receipt that re-verifies against the current snapshot exposes the closed `authoritative_final: success|failure|blocked|partial|not_evaluated` verdict. Derive, report, dashboard, closeout, and later cycles must reject a missing, stale, mismatched, or uncommitted receipt. Required actual-body truth, report convergence, artifact class/freshness/lane, consumer context, or verifier completeness at `not_evaluated` prohibits `advanced`.
- Treat finalization durable state as a closed owner contract. A typed operation must match an owner-registered `target_ref`/`target_kind`/`payload_schema_id`/operation/recovery tuple and the registered closed payload-body validator at build and validation time; a known schema ID never authorizes arbitrary fields. Nested durable metric observations must not retain raw non-scalar values or unbounded strings: persist only an opaque value reference bound to a full digest and bounded type/cardinality summary, or a bounded scalar/enum. `no_durable_state_change` requires the registered validation reason plus non-empty, attempt-bound owner observations whose before/current revision and state digest are exactly unchanged and whose per-target receipts are content-bound. An arbitrary reason, reason-only hash, target-ID list without observations, fabricated owner, rehashed state change, unregistered target, or missing/mismatched evidence cannot finalize. Preserve exact replay, target CAS, and pending-conflict behavior described in [cycle-artifacts.md](references/cycle-artifacts.md#finalization-durable-state-contracts).
- On a valid-candidate compare-and-swap conflict, preserve the attempt through the ledger's immutable pending-finalization journal. Do not overwrite the current pointer or count the pending record as progress. Feed active `pending_conflict` records into derive, allow only `finalization_conflict_recovery` or `finalization_state_reconcile`, and resume normal successor selection only after an immutable `merged` or evidence-bound `retired` resolution.
- Consume the verified axes without collapsing their scopes. A deterministic `task_acceptance_verdict: pass` may close the bounded task in the existing task index even when reviewer availability leaves artifact semantics `not_evaluated` or goal readiness blocked. Preserve that alias as completed current history and non-executable; derive only a distinct successor from finalized residual evidence. If no distinct successor evidence exists, keep the completed alias in the existing terminal-wait/current-history state rather than reselecting the same task or fabricating a terminal/global-completion claim.
- Before starting a full cycle from a terminal wait state, compare `(terminal_outcome_family_key, blocker_signature, input_state_fingerprint, authority_state_fingerprint, external_state_fingerprint)`. An identical current-version tuple reuses the existing terminal wait record instead of producing another full cycle; a stale `material_delta` flag cannot reopen it. Do not latch when loopback reports self-resolvable or unverified residuals. Require one atomic seal/registry/pack/index receipt before the single reopen, and keep an incomplete reopen out of cycle initialization.
- While terminal wait is unchanged, resolve the authority-settled current baseline and run `selection-tick` only; do not assign proposal agents or initialize a cycle. Treat wake-predicate and minimum-delta IDs as policy labels. The executable rule is fixed: a changed manifest must contain either an added/content-changed exact-premise row or a changed row in a baseline-bound evidence class. Exact premise and effective authority rows remain sticky when omitted; identical premise replay, including replay accompanied only by nonmaterial drift, is `no_op`. For autonomous re-entry, require a consumed, artifact-verified `validated_exact_subject_premise_receipt_v2` bound to the current wait; a raw digest or structural v1 receipt does not prove workspace artifact identity, freshness, canonical ownership, first-failing-invariant fit, or source-separated/replayable evidence. A valid `selection_required` tick opens the existing derive selection boundary, never a full cycle. If derive returns to terminal wait, reopen the three durable lens projections and canonical synthesis-output projection, seal `derive_selection_synthesis`, bind it to the exact trigger in a preliminary decision, and bind persisted trigger plus preliminary decision in a durable `selection_decision_receipt`. Treat the activated predecessor as `A`, the `selection_required` trigger as `B`, and the input-stable acknowledgement/rebase as `C`: require `B` to descend from `A`, `C` to descend unchanged from `B`, and the direct final derive's `terminal_wait.last_selection_receipt` to bind the reopened receipt. Caller-authored IDs and syntactically valid hashes are insufficient. Publish only a direct full final derive result that embeds `C` and the receipt-bound analysis; a `{ "result": ... }` wrapper or preliminary decision cannot be `source_derive`. Materialize the content-addressed non-circular terminal authority subject from that final result, `C`, and the exact predecessor snapshot, then use the authority-settled terminal-wait baseline owner so later ticks can auto-discover it. Before consuming a selected successor, require `selection-publication status` to be clear; publish `task.md` and any archive/index/pack projections through one exact-before/after journal with `task_alias` last, or forward-recover the pending journal. A prepared or partially written publication is non-authoritative, physical deletion is not representable, and target drift blocks rather than overwrites foreign bytes. Use [selection-reentry-and-publication.md](references/selection-reentry-and-publication.md) for the bounded commands and schema.

## Completion And Reporting

Finalize validation scope from governance's actual changed files, snapshot the task-state index, and run `$validate-task-completion` before issue tracking, schema pre-derive reconciliation, next-task derivation/promotion, implementation commit, final report, or closeout commit. Validation must report `validation_verdict`, `progress_verdict`, `progress_axes`, `final_candidate: true`, and the attempt identity when it evaluates a current attempt. Before leaving the existing `validate` boundary, finalize the candidate with the cycle-ledger helper and re-verify its `cycle_finalization_receipt`; do not append completion-like current state or enter any consumer on candidate evidence alone. `progress_verdict: advanced` requires `terminal_outcome_changed=true` or strict changed-and-semantic observed output delta, and the finalized authoritative projection must preserve that downgrade-only result.

When Part O/O2' `goal_axis_map` evidence is present at close, preserve per-axis `axis_delta` and `axis_stall_streak` for `$render-cycle-dashboard` and `$derive-improvement-task`. If an axis is stalled at the adapter-owned threshold, the next derivation must prefer direct axis advancement or record the terminal/user-escalation blocker. Missing hook evidence is a no-op.

Call `$manage-implementation-issues` for the just-validated current task before schema pre-derive or `$derive-improvement-task`. Only after derivation, schema reconciliation, and final indexing may `$repo-change-commit` finalize coherent implementation/workflow changes in a Git worktree; a raw `git status` check is not a substitute. A `partial` verdict can still produce a checkpoint commit when the changed set is coherent and blockers are named.

Use [cycle-report-template.md](references/cycle-report-template.md) for the Korean final report. Use `complete_verified` only when the validator's complete/pass vocabulary (`complete`, `passed`, `pass`, or `success`) is paired with `progress_verdict: advanced` and no blocker remains; otherwise use `not_complete`.

After report/dashboard artifacts are written, run the closeout commit phase when applicable. Do not regenerate closeout/dashboard/report/recheck artifacts merely to reconfirm terminal quiescence or G2 terminal escalation.

## Failure Handling

Preserve evidence and route remediation to the owning skill:

- Missing `task.md`: complete the separate bootstrap transaction, then start a fresh normal cycle at task-bound adapter scan; never normalize acceptance from task-absent context.
- Governance blocker: stop before unsafe execution and keep task_miss/index evidence.
- Execution failure: log through `$run-task-code-and-log`, collect scalar-safe failure autopsy when possible, finalize validation scope, snapshot the pre-validation index, then validate as `partial` or `failed`.
- Running execution: log monitor/stop details, index the active run, defer final-output-dependent derivation or completion claims.
- Long-running branch: record `run_id`, owner task, launch cycle, body-free `command_argv`, workdir, output/log/checkpoint paths, heartbeat, monitor/stop commands, expected completion signal/artifacts, and remaining validation before leaving the process running. Defer final-output-dependent review/loopback and permit only `partial` handoff completion validation plus non-closing issue tracking. A pending run cannot produce `passed`, `advanced`, schema/derive promotion, task-pack promotion, or unrelated next-task derivation until monitor/harvest resolves the run or records a terminal/user-escalation blocker.
- Long-running branch with Part M evidence: keep launch manifest anchors, target scale, harvest-gate preflight status, risk-acceptance flag, terminal disposition policy, and reharvest path in the run packet so monitor and harvest consumers do not derive expectations from the current task, current lane, capped samples, or stale validation contracts.
- Review, validation-set, schema, issue, adapter, or Git failure: record the concrete blocker and continue only through safe downstream evidence-preserving steps.
- ID audit high-severity findings: do not delete artifacts or report completion unless `$validate-task-completion` resolves or downgrades them.
