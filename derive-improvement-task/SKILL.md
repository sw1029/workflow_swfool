# Derive Improvement Task

## Overview

Use this skill to turn current goal context, repository evidence, previous misses, and candidate ideas into the next concrete `task.md`.

When long-range ordering is needed, this skill may also create or update a task pack under `.task/task_pack/`. The pack is planning state only; the final executable output remains one active `task.md`. Treat pack updates as a transaction: choose exactly one `pack_disposition`, mutate the JSON queue with evidence, refresh the Markdown render, and promote at most one item into `task.md`.

The workflow is intentionally agent-heavy: use `$inspect-repo-with-agents` for critical goal/convention-alignment analysis, agent-based `.task/task_miss` analysis, one issue goal-fit agent from `$manage-implementation-issues`, three parallel improvement-analysis agents, one synthesis agent, and `$record-agent-work-log` to preserve the previous `task.md` as `past_task` before replacing it.

Use `$manage-task-state-index` for task, candidate, miss, past_task, log, audit, and validation IDs whenever possible. If no ID context exists, continue deriving the improvement task and report that indexing was skipped.

Use `.agent_goal/goal_schema_contract.md` and `.schema/` or `.contract/` contract context from `$manage-schema-contracts` when present. Schema, module, script, version, and causal compatibility contracts are planning evidence, not optional documentation, when the selected task can change interfaces or data artifacts.

Use `$manage-agent-authority` to resolve authority goal truth for task selection. Its result should describe the current agent's effective operating permissions and any narrower project-specific policy for API calls, external service use, direction freedom, strictness, implementation-first versus validation-first priority, output/artifact confirmation, quality review, conservative implementation, and escalation. If the file is absent, consume `authority_policy: default_current_agent_permissions`: current sandbox, approval, user/developer/system instructions, and no inferred extra API/network/destructive authority. This policy may constrain or rank task options, but it cannot override higher-priority instructions or grant missing capabilities.

Use `$manage-external-advice` when `.agent_advice/active/*.md` exists or the caller passes an advice packet. Active advice is non-GT planning evidence: it can steer task choice, design notes, schema planning, and validation expectations, but it must be listed as `used_advice`, kept out of `.agent_goal`, and rejected or deferred if it conflicts with GT, authority, current user instruction, or repository facts.

Use `.issue/` and GitHub issue mirrors from `$manage-implementation-issues` when present. Active issues are planning evidence only after a first-pass issue agent classifies whether each issue fits the goal.

Use `.task/task_pack/*.json` when present. The JSON queue is canonical; `.md` files beside it are user-language renders. Read [task-pack-workflow.md](../orchestrate-task-cycle/references/task-pack-workflow.md) when a caller supplies a task-pack packet or `.task/task_pack/` exists.

If `task.md` is absent, run this skill in initial-init mode: derive the first actionable `task.md` from goal context, repository evidence, issues, task_miss, and candidate_task without requiring or archiving a previous task.

## Selection Efficiency Policy

Keep task derivation conservative, but avoid selecting another task whose main effect is only to add one more small no-live check.

- Prefer blocker-state-transition tasks: select work that moves a blocker from one meaningful state to the next, such as `missing evidence -> intake path proven`, `intake path proven -> bounded preflight`, `bounded preflight -> source-backed validation`, or `schema drift -> compatibility proven`.
- Merge adjacent no-live micro-contracts when they share the same target script/module, issue blocker, prerequisite chain, promotion boundary, and validation surface. A merged task may still be narrow, but it should close a coherent boundary as one cycle.
- Classify recent completed tasks by progress, not only validation. Use `advanced` when the task unlocked a new execution or goal state, `safety_only` when it only proved fail-closed/no-live safety, `no_progress` when it added no meaningful blocker movement, and `regressed` when it weakened or broke a required state.
- If the last two completed tasks are `safety_only` and the same blocker remains open, the selected next task must supply or validate the missing evidence path, perform a bounded source-backed run/preflight, consolidate remaining no-live boundaries into one batch task, or explicitly stop with a blocker explaining the missing external input or authorization.
- If no active task pack exists and a known 2-5 step sequence would prevent repeated myopic derivation, create a bounded pack and promote only the first safe item into `task.md`.
- If an active task pack exists, consider the next pack item before standalone candidates. Promote it unless current evidence requires insertion, reordering, skipping/excluding, superseding, standalone bypass, or terminal blocking.
- If a candidate or pack item comes from a measurable user/advice/issue directive, preserve the original target through `scope_fidelity` and copied acceptance. Do not make the item `goal_productive` by replacing the original target with a weaker pilot/plan/slice criterion.
- If a measurable directive must be narrowed for one cycle, set `scope_fidelity.narrowed=true`, record `narrow_reason`, and create an open residual item before the narrowed item can be promoted or consumed.
- If normalized acceptance or a caller packet provides `acceptance_envelope_contract`, do not select a task slice whose planned execution envelope is below the adapter-owned `envelope_floor` as `goal_productive`. Select envelope expansion, constraint relaxation, explicit descope with residual scope, terminal-block, or user escalation according to authority; do not lower the measurable target to fit the slice.
- If normalized acceptance, task-pack scope fidelity, or active advice provides `acceptance.quantifiers` or `evidence_kind`, preserve them in the selected task or pack mutation. A criterion tagged `live_run` requires a fresh run id after item creation; do not select or consume a derived-artifact, code-contract, or report-only substitute as complete. If narrowing or substitution is unavoidable, set `acceptance_diluted=true`, preserve the original criterion, and keep an open residual item.
- If normalized acceptance or validation-set evidence provides `acceptance_scenarios`, preserve premise-class -> expected-terminal-state coverage. `scenario_uncovered=true` selects validation-set/fixture/live-run supply before close. `acceptance_inversion=true` selects code/contract repair; do not select another green-test-only confirmation task.
- If normalized acceptance or a caller packet provides `acceptance_verifier_contract` and the required verifier is `not_evaluated`, do not select a task that consumes the measurable target as `goal_productive`. Select verifier-hook implementation, metric/oracle correction, explicit descope with residual scope, terminal-block, or user escalation. A missing required verifier is `unverifiable_acceptance_contract`, not a successful gate.
- If normalized acceptance or a caller packet says a measurable target depends on a gate's required hook and that hook is missing or `not_evaluated`, use the same `unverifiable_acceptance_contract` path. Select hook-supply/adapter-wiring work, explicit descope with residual scope, terminal-block, or user escalation; do not consume the target through gate fail-quiet.
- If a caller packet provides `residual_gap_policy`, compare the residual gap's marginal value against explicit `descope-with-residual` plus the next capability-ladder rung. When `residual_gap_ratio` is below the adapter-owned threshold, label another same-gap repair as `marginal_repair`, keep it behind residual descope and the next rung, and preserve the original target as open residual scope. If cycle-efficiency/profile evidence supplies `cycle_fixed_cost`, compare value per cycle cost; if cost evidence is absent, use denominator `1` and keep existing F3. If the hook is absent, fail quiet and keep existing selection rules.
- If a caller packet provides `outcome_variance` with `predetermined_unreachable=true` or `floor_edge_envelope=true`, route to contract revision, interval/ratio/intersection acceptance, envelope expansion, explicit residual descope, terminal blocker, or user escalation instead of retry.
- If a task or pack item carries an output-derived scalar expectation, require `expectation_anchor` to be compared against the current `designated_baseline` before promotion. If `expectation_lineage_stale=true`, select expectation rebaseline against the designated surface or fail closed before live execution; do not spend a run discovering a statically knowable mismatch.
- If `expectation_anchor_missing=true`, fail quiet but keep the selected work from claiming lineage-verified expectation evidence. Prefer a small rebaseline or anchor-supply item when the missing anchor would otherwise make the run predictably fail.
- If a caller packet or pack item carries `parity_unverified=true`, missing `parity_axes`, or any parity axis with status `unknown`, do not select final adoption or baseline promotion. Select parity-axis resolution, an explicitly provisional comparison, residual descope, terminal blocker, or user escalation.
- If an adoption packet uses `majority_vote_adoption=true` without `adoption_axis_classification`, downgrade adoption to provisional and select axis-classification work before final adoption. A failed `gating` axis always blocks adoption regardless of tradable-axis counts.
- If `measured_but_disqualified=true`, preserve the measured candidate as evidence but do not select work that promotes it as canonical baseline or accepted winner unless a later packet clears the failed gating axis.
- If `resolution_downgrade=true`, treat the pass as lower-resolution evidence only. After repeated downgrade for the same contract, select resolution restoration or explicit contract revision instead of another surrogate-consuming comparison.
- If `report_key_divergence=true`, select report/schema/sync repair or terminal/user escalation. Do not select a task that consumes the divergent report as pass, close, baseline, comparison, or high-water evidence.
- If `pass_on_stale_lane=true`, do not select work that consumes the pass as current-lane capability, adoption, comparison-winner, or next-rung evidence. Select current-lane rerun/revalidation, residual descope, terminal blocker, or user escalation.
- If `decision_metadata_revision=true` or `stale_measurement_artifact=true`, classify the update as metadata/governance only. Select fresh current-lane measurement, a no-impact proof for the upstream production-contract change, explicit residual scope, terminal blocker, or user escalation before consuming measurement/adoption work.
- If `axis_starved_by_missing_producer=true`, select producer-supply work for the gating axis before selecting another verifier, guard, consistency check, or report for that axis. Verifier-like work over that axis is not goal-productive until the producer path exists and fires.
- If `portfolio_quota_exceeded=true` with restrict mode, select only producer, envelope, long-run, descope-with-residual, terminal blocker, or user escalation until the verifier-like to producer-like ratio recovers. If quota evidence is warn-only, preserve it but do not restrict selection by itself.
- If `unreachable_within_cycle=true`, select long-run launch with monitor/harvest plan, throughput improvement with measured throughput target, explicit residual descope, terminal blocker, or user escalation. Do not select another small smoke rerun as `goal_productive`.
- If a long-run launch candidate has Part M `harvest_contract_preflight` findings, select harvest-gate repair/mitigation before launch unless `harvest_risk_accepted=true` is explicit. Do not select silent risky launch when non-degradable checks are lane-, scale-, or contract-incompatible.
- If terminal evidence shows `high_cost_artifact=true` and verifier/governance failures dominate, select quarantine-preserving gate repair followed by reharvest before a new full rerun when `reharvest_path.available=true`. A rerun before attempted or terminal-blocked reharvest is `rerun_before_reharvest` and is not goal-productive.
- If a predicate/directive pair reports `mutually_unsatisfiable_contract=true`, select same-task predicate/directive reconciliation, explicit residual descope, terminal blocker, or user escalation before any task that consumes either side as valid.
- If a result consumer reports `sample_as_universe_misuse=true`, select full untruncated collection supply or contract revision to sample-only consistency before pass/close consumption.
- If `deliverable_stale=true`, select regeneration/materialization plus revalidation before any task that consumes the repaired producer or verifier path. When `reharvest_path.available=true`, use that opaque route instead of inventing a command; do not count this once as M2 and again as N2.
- If `feature_regressed_artifact=true` or `feature_regression_count>0`, force landed-feature inheritance repair above any task that consumes the regressed representative artifact. The selected work must restore adapter-authorized opaque feature classes, explicitly descope the missing inheritance with residual scope, terminal-block, or user-escalate. Missing `landed_feature_inventory` is fail-quiet and must not invent feature classes. Do not double-count P1 as Part L stale-lane consumption or N2 deliverable staleness.
- If `reason_code_rank` evidence is supplied for a fail-close item, rank repairs for `primary` reason codes above repairs for `secondary` reason codes and respect any `precedence_edges` before choosing the next task. If a candidate repairs only a secondary gap while a primary failure remains actionable, mark it `misdirected_repair` and keep it below the primary repair. If the hook is absent, fail quiet and keep existing selection order.
- If `reason_to_attempt` evidence or an anti-loop packet maps fail/degrade `reason_codes` to exact `root_key` attempt counts, do not terminal-block or seal while any primary reason has `attempt_count==0`. Select the untried reason's exact root-key repair as the next `goal_productive` task, or user-escalate only if authority/input evidence blocks that repair. Similar, adjacent, renamed, or suffix-varied root keys do not count as attempted. If the hook is absent or malformed, preserve existing selection semantics and keep `coverage_unverified_vs_reason_codes` as warning evidence.
- If `self_resolvable_without_authority=true` or Q1 probe evidence reports `self_resolvable=true`, select existing-capability rewiring or deterministic input-generation work before terminal/user escalation or another frozen-input consumption task. Do not treat this as a positive user-supplied input delta; it is an in-scope engineering route for the same missing input.
- If `required_freshness_class=fresh_producer_execution` and `producer_executed=false`, do not select work that consumes the run as fresh producer evidence. Select producer re-execution, input-generation refresh, explicit descope with residual scope, terminal blocker, or user escalation; preserve reprocessing scalars only as `derived_artifact` evidence. If `condition_unsatisfiable_for_input_generation=true`, route to input-generation refresh or the same descope/terminal/escalation choices instead of ordinary `not_applicable` consumption.
- If `diminishing_reprocess=true` or frozen-input evidence shows `frozen_input_consumption_streak` at the adapter-owned threshold with `generation_time_defect_dominant=true`, rank producer re-execution or input-generation refresh above another post-processing task. If producer re-execution/input refresh is outside authority, emit terminal/user escalation with the missing permission or input named instead of deriving another post-processing item.
- If `constraint_forced_vacuity=true`, do not select another metric/oracle/rung task as `goal_productive` for the affected metric. Use this flow: metric passed -> `substance_density` absent means fail quiet; density below adapter floor means N1; `vacuity_cause=production_constraint` selects persistence-policy reconciliation, while `vacuity_cause=missing_producer_capability` selects producer supply. Tautological metrics stay under G-OENV.
- If `anonymization_theater=true` or `mutually_unsatisfiable_persistence_contract=true`, select persistence-contract reconciliation before consuming the storage-policy task as progress. Prefer an ability-preserving opaque-id plus transient-merge-key alternative when the packet offers it; otherwise select explicit residual descope, terminal blocker, or user escalation.
- If `hook_demand_unaccounted=true`, select repair of hook-demand accounting before creating another validator/detector for the same missing input. If `adapter_hook_demand` rows reached `hook_demand_threshold`, select hook wiring or `adapter_hook_batch_supply` above new validation surfaces unless the task records observability without the hook, descope, terminal blocker, or user escalation.
- If `same_module_repair_streak` reaches the adapter-owned Part O/O1' threshold with `repair_shape=special_case_addition`, do not select another same-lineage special-case repair as `goal_productive`. Select the adapter-indicated structural transition, explicit observability/descope, terminal blocker, or user escalation; if `repair_file_lineage` is absent, fail quiet and keep existing distinct-root and same-family rules. Do not double-count O1' as a G-series distinct-root finding.
- If `goal_axis_stall=true` or an `axis_stall_streak` reaches the adapter-owned Part O/O2' threshold, rank direct advancement of that axis above further preparation, gate, metric, lineage, or report work for the same axis. If direct advancement is impossible, write the missing input/authority/evidence as a terminal blocker or user escalation. Missing `goal_axis_map` is fail-quiet and must not change selection.
- If `basis_overclaim=true`, select basis-compatible measurement, metric-basis repair, contract downgrade with residual scope, or terminal/user escalation. Do not use the affected metric as independently verified progress.
- If a qualitative review supplies nonzero `surface_field_defect_matrix`, select producer/field repair, residual descope, terminal blocker, or user escalation before consuming the review pass for affected fields.
- If caller packets provide `pass_with_unobserved_axes=true`, `goal_axis_completeness_gate.evaluation_status=fail`, or nonempty `unobserved_goal_axes`, do not consume the qualitative review as pass for the affected measurable goals. Select adapter axis supply, an output-producing task that supplies the missing observable axis through the adapter, explicit descope with residual scope, terminal-block, or user escalation.
- If anti-loop evidence shows a generation-dependent raw family/root key or trace-only `legacy_family_key`, do not treat proximate label/task/advice/pack/cycle/run changes as a new blocker family. Select based on the effective adapter-collapsed root plus dominant parameter or terminal-outcome family.
- If recent evidence repeats the same normalized blocker signature, do not select another narrowing or handoff task unless a new input kind, authority change, external-state change, or safe pack item changes the blocker state.
- If recent evidence repeats the same suffix-normalized `root_key` or `semantic_signature`, treat it as the same blocker family even when target-surface, run directory, timestamp, `after-*`, date, sequence, or `vN` suffixes changed. Do not select another non-terminal task in a sealed semantic family without a supplied input artifact/domain delta, authority change, or external-state change.
- If the caller supplies `feature_symbol_gate`, use it as the observed-over-declared loop signal. Treat repeated symbols with `observed_output_class` of `metadata_only` or `terminal_record` as the same workflow attempt even when labels, task names, run directories, signatures, or version suffixes changed.
- If `feature_symbol_gate.hard_stop_required=true`, select goal-productive work, consolidation, terminal-blocked, or user escalation. Do not satisfy this gate with another metadata-only handoff, renamed command family, or task that relies only on self-declared `produced_domain_delta`.
- If the caller supplies an output-delta gate, use `effective_progress_kind` instead of a self-reported `progress_kind`. A task is `goal_productive` only when repository-declared output layer paths changed and `semantic_progress=true`, validated positive evidence changed, or another non-sidecar artifact reduces goal distance. Metadata-only, workflow-only, reconciliation-only, sidecar-only work, and non-empty counts without semantic change remain `governance_only` even when validation passes.
- If run or producer artifacts contain `observed_producer_claim`, treat it as trace-only self-report. Do not use producer-declared `goal_productive`, `advanced`, `effective_progress_kind`, or `produced_domain_delta` as the selected task's progress truth source unless adapter-recomputed or strict output-delta evidence independently supports it.
- If `anti_loop_progress_gate.evidence_provenance_gate.attested_only_movement=true`, treat that movement as no high-water movement. Do not use producer-attested scalar movement to satisfy a hard `goal_productive` requirement, reset a stall, or avoid C4 forced retargeting.
- If `anti_loop_progress_gate.verification_source_separation_gate.independent_source_separation_status=missing|overlap|blocked` or `independently_verified_downgraded_fields` is nonempty, treat affected `independently_verified_fields` as attested. Do not select `goal_productive` from zero-disagreement or independent verification claims unless verification inputs are disjoint from verified artifacts or the adapter marks the axis `self_grounded`.
- When an output-delta gate includes `observed_output_class`, prefer the observed value over declared fields. Only `observed_output_class: node_edge_delta` resets no-delta loop counters.
- For evidence-family tasks with `positive_input_delta_required: true`, require at least one newly introduced input kind and a supplied positive input delta before claiming progress. The supplied delta must be a non-empty artifact path in `supplied_input_artifact_paths` or `produced_domain_delta=true`.
- For metadata-only loop evidence, require a positive output delta or supplied input artifact before treating the next evidence-family task as progress. New wording, a new timestamp, a renamed input kind, or another metric-only cohort is insufficient.
- If the caller supplies `qualitative_review_packet.progress_cap=governance_only` because semantic output is not ready, placeholder events were found, or surface entities are suspected, preserve that cap unless independent strict output-delta evidence proves changed-and-semantic primary-output progress.
- If the caller supplies `gt_constraint_conflict_packet.status=block`, do not select a task that inherits the contradictory prohibition. Select a task that resolves the task-level contradiction, set `resolves_gt_constraint_conflict=true` or `selected_task_kind: gt_constraint_conflict_resolution`, or emit `selected_task_source: terminal_blocked` with the required user decision.
- If run evidence, failure autopsy, or the loop-breaker packet includes `classification: self_inflicted_gate_defect`, do not select another same-gate environment recheck. Select a gate-contract/code correction task that makes the gate satisfiable, switch to the recorded `alternative_evidence_source` when one exists, or emit `selected_task_source: user_escalation` with `required_missing_input.kind: gate_contract_fix_approval`.
- If run evidence includes `runtime_config_echo.config_overrides` with `config_origin=code_default` and the override explains a local cap, incompatible default, or self-imposed missing evidence shape, prefer a gate/default correction task or user escalation over another opaque retry.
- If run evidence includes `run_disposition=candidate_degraded`, treat the produced artifact as preserved quality-miss evidence only. Do not select a task that promotes it to canonical baseline or closes the item unless independently verified axes and source-separation evidence allow that specific consumption.
- If run evidence includes `command_provenance_missing=true`, do not select comparison, A/B, baseline, or reproduction work that depends on that run. Select full-argv rerun/provenance repair or exclude that run from those claims.
- If blocker evidence shows repeated `blocker_opacity=true` for the same gate, select blocker-contract repair that emits violated relation, observed values, expected relation, or minimum input delta before another opaque recheck. If the blocker describes an implicit relation across multiple caller inputs or `authorization_contract_repair_candidate=true`, prefer a contract repair that exposes the related abstract keys and, where feasible, replaces the hidden relation with a named single authorization input.
- If `anti_loop_progress_gate.repo_owned_source_roots_status=provided` and a root-cause ledger entry has `repo_owned_source_refs` or `self_report_rejected_fields`, treat that entry as provenance-hardened actionability. Do not terminal-block merely because the same producer reported `local=false`, `in_scope=false`, or `actionable=false`.
- If `failure_autopsy.gate_selfcheck.status=warn_missing_repo_owned_confirmation`, use it as a warning only. Require loopback/actionability provenance, adapter confirmation, or an explicit caller flag before selecting a self-inflicted gate-fix task from that self-check alone.
- When the caller supplies `goal_distance_gate.requires_goal_productive_next`, select `progress_kind: goal_productive` work or emit `selected_task_source: terminal_blocked`. Do not satisfy this gate with workflow sidecars, hash renames, or another governance-only reconciliation task.
- When the caller supplies `root_axis_gate.autonomous_retarget_disabled=true`, select `progress_kind: goal_productive` work or emit `selected_task_source: terminal_blocked`. A new `semantic_signature`, new wording, new version suffix, or provider-neutral retarget on the same `root_axis`/`root_key` is not enough.
- When the caller supplies `effective_allowed_dispositions`, choose the selected task disposition from that list only. Treat gate `allowed_dispositions` as an intersection already computed by loopback/orchestration, not a menu to union. If `disposition_intersection_basis` includes `allowed_task_kinds`, emit `selected_task_kind` and keep `goal_productive` inside that set; a matching label with a different task kind is `governance_only` or blocked. If the only effective options are `terminal_blocked` and `user_escalation`, do not write another standalone consolidation task.
- When the caller supplies `consolidation_streak >= consolidation_streak_cap`, do not select another consolidation task unless independent strict changed-and-semantic primary-output evidence resets the streak.
- Classify the selected task as `progress_kind: goal_productive` or `progress_kind: governance_only`. After two governance-only cycles, select goal-productive work or terminal-block instead of deriving another narrowing/handoff task.
- After two metadata-only cycles, select `resume_primary_output`, bounded source-backed run/preflight, safe failure root-cause repair, or terminal-block. Do not derive another measurement-only task unless a repository output-delta contract reports `produced_domain_delta: true` or the task directly creates the missing primary output path.
- If `anti_loop_progress_gate.measurement_progress=true`, count it as governance-only instrumentation unless `measurement_progress_allowed=true`, `coverage_quality_delta_gate.quality_delta_pass=true`, `coverage_quality_delta_reconciliation_gate.status!=block`, and `substance_delta_gate.substance_delta_pass=true`. Do not use the exemption when `measurement_progress_streak_for_root_key > measurement_streak_cap`, `measurement_progress_streak_for_root_family > measurement_streak_cap`, when the check/oracle ID is already known, when validator disagreement, R-GCOV disagreement, or validator integrity blocks the packet, or when G-COV/G-SUBSTANCE fails.
- If `anti_loop_progress_gate.substance_delta_gate.status=missing` or `substance_delta_pass=false`, do not promote validator/oracle existence, scalar metrics, or capability-ladder observations as `goal_productive` unless strict changed-and-semantic primary-output evidence exists. Select substance-producing implementation/correction work, terminal-block with the missing adapter/output evidence, or user escalation.
- If `anti_loop_progress_gate.constraint_forced_vacuity=true`, do not promote the affected metric as `goal_productive` even if it passed. Route production-constraint vacuity to N3 persistence-policy work and missing-producer-capability vacuity to L3 producer supply; if `substance_density_unchecked=true`, keep existing selection semantics and preserve the unchecked hook as fail-quiet evidence.
- If `anti_loop_progress_gate.adapter_wiring_defect=true`, select `selected_task_kind: adapter_wiring_fix` or `adapter_load_fix` as the next goal-productive task before adapter creation, adapter strengthening, or another domain micro-repair. If authority or local context prevents the correction, emit `terminal_blocked` or `user_escalation` with `adapter_path`, `adapter_expected_path`, and evidence paths.
- If `anti_loop_progress_gate.adapter_mandate_required=true`, select adapter registration or adapter strengthening as the next goal-productive task before another domain micro-repair. If authority or local context prevents adapter work, emit `terminal_blocked` or `user_escalation` with `adapter_contract_unmet`, `adapter_missing_streak`, and evidence paths.
- If `anti_loop_progress_gate.hook_supply_required=true`, apply the `$orchestrate-task-cycle` `references/workflow-routing.md` derive-row rule for `hook_supply_required` verbatim: another same-family or same-gap goal-productive task is invalid until the selected task supplies the `demanded_hooks`, records why the target is observable without the hook, descopes with residual scope, terminal-blocks, or user-escalates. If `demanded_hooks` has two or more entries, prefer `selected_task_kind: adapter_hook_batch_supply`.
- If `anti_loop_progress_gate.hook_demand_unaccounted=true`, select registry/accounting repair for decision-relevant fail-quiet skips before deriving a new detector or verifier that depends on the same missing adapter input.
- If `anti_loop_progress_gate.cumulative_goal_distance_stalled=true`, first select `anti_loop_progress_gate.forced_selected_task.selected_task_kind` when `chain_stall_forced_retarget_gate` provides an actionable forced option. Otherwise select only `terminal_blocked` or `user_escalation` unless `adapter_mandate_required=true` is the preceding forced adapter task. If `untried_veto_overridden_by_chain_stall=true`, do not promote another distinct untried hypothesis merely because its label or terminal outcome is new.
- If `anti_loop_progress_gate.root_dominant_parameter_key` is present, treat distinct root-cause and stall evidence as keyed by collapsed root plus that parameter. Do not promote another untried repair merely because proximate labels, suffixes, target-surface wording, or rung names changed.
- If `anti_loop_progress_gate.failure_surface_stage_gate` reports `terminal_classification_stage_contradiction=true` or `terminal_classification_invalid_for_counting=true`, do not count the failure as terminal or close the family. Select terminal-classification-stage repair, same-input/input-contract repair, terminal-block, or user escalation with the conflicting stage evidence.
- If `anti_loop_progress_gate.same_input_contract_gate.same_input_contract_violation=true`, do not compare or close same-family failures across mismatched input sets. Select a same-input contract repair or instrumentation task before another counted repair.
- If `anti_loop_progress_gate.diagnostics_unavailable_gate.instrumentation_supply_required=true`, the forced option set must include instrumentation supply. ## Execution Environment Selection

- Prefer an environment section copied from the previous `task.md` only when it names a runnable environment and still matches the selected task's language/dependencies.
- If environment information is absent, stale, or too vague for a Python task, invoke `$find-local-python-envs` with dependencies inferred from the selected task, repository manifests, tests, imports, notebooks, and candidate evidence.
- Choose the highest-ranked usable environment in this priority order:
  1. Conda: `conda run -n <env> ...` or a conda prefix command.
  2. Venv/virtualenv: `<repo>/.venv/bin/python ...`, `venv/bin/python ...`, or a documented project venv.
  3. Local/system Python: `python`, `python3`, or an absolute interpreter path discovered on PATH.
- Do not install packages or create environments during task derivation.
- If no candidate satisfies dependencies, still write the task but make the unresolved environment a visible prerequisite in the top section and validation plan.

## Selection Rules

Prefer the improvement that:

- Best advances `.agent_goal/final_goal.md` success criteria.
- Respects `.agent_goal/conventions.md`; if a convention conflict exists, the selected task explicitly resolves or documents it as a blocker rather than proceeding as if compliant.
- Fits the documented architecture and theory, or explicitly updates gaps discovered by analysis.
- Incorporates active `.agent_advice` only when it is compatible with current user direction, GT, authority, and repository facts; otherwise explicitly defers or rejects it.
- Preserves documented `.schema` and `.contract` compatibility, or makes the necessary schema/contract version update explicit in the task.
- Respects the `$manage-agent-authority` result, including `.agent_goal/agent_authority.md` when present or the `default_current_agent_permissions` fallback when absent, including API/external-call policy, direction freedom, strictness, implementation/validation priority, and conservative implementation constraints.
- Satisfies `.agent_goal/goal_schema_contract.md` minimum fields, mandatory rules, target module/script guidance, and causal-map expectations when schema/contract surfaces are involved.
- Addresses high-severity `.task/task_miss` patterns.
- Fits active implementation issues that the issue agent classifies as goal-aligned, or explicitly explains why a misfit/stale/duplicate issue should not drive the next task.
- Can be validated with concrete tests, commands, metrics, or review evidence.
- Is `goal_productive` when the caller's goal-distance gate requires it: expected to produce changed-and-semantic goal-relevant output, quality evidence, source-backed validation, or another non-sidecar artifact that reduces goal distance.
- Uses a `selected_task_kind` allowed by `disposition_intersection_basis.allowed_task_kinds` whenever gates constrain `goal_productive` to specific correction classes.
- Is not classified as `goal_productive` merely because it writes workflow metadata, sidecar metrics, schema records, task-state records, non-empty counts, lineage, gap reports, or measured absences without changed-and-semantic repository-declared output delta or validated positive evidence.
- Does not consume `pass_with_coupled_verifier`, `producer_attested_fields`, or `attested_only_movement` as progress truth.
- Does not consume contradictory terminal classifications, same-input mismatches, diagnostics-unavailable repeats, non-disjoint independent verification, or missing envelope thaw items as ordinary `goal_productive` progress.
- Does not consume supplied-but-unexercised instrumentation, uncovered scenarios, acceptance inversion, missing command provenance, repeated opaque blockers, stochastic infeasible contracts, first-fire instrumentation credit, live-run criteria satisfied by derived/report/code-contract evidence, guard-stacking hardening past cap, or `candidate_degraded` artifacts as ordinary `goal_productive` progress.
- Does not consume stale output-derived expectations, unverified comparison parity, failed gating adoption axes, lower-resolution surrogate evidence, or divergent duplicate report keys as ordinary `goal_productive`, close, baseline, comparison, adoption, or high-water evidence.
- Does not consume stale-lane passes, stale decision measurements, producer-starved gating axes, restrictive verifier-like portfolio excess, cycle-unreachable smoke reruns, basis-overclaimed metrics, or unresolved surface-field defect counts as ordinary `goal_productive`, close, baseline, comparison, adoption, next-rung, or high-water evidence.
- Does not consume risky harvest preflight, destructive high-cost non-safety disposition, rerun-before-reharvest, mutually unsatisfiable predicate/directive contracts, or truncated/sample collections as ordinary `goal_productive`, close, baseline, comparison, adoption, next-rung, or high-water evidence.
- Does not consume `pass_with_unobserved_axes` or a failed `goal_axis_completeness_gate` as qualitative pass evidence for measurable goals.
- Treats below-threshold residual gap repair as `marginal_repair` unless adapter evidence shows its marginal value, including value per cycle cost when supplied, exceeds explicit residual descope and the next capability rung.
- Treats generation-dependent family/root keys as trace-only and uses effective root-family/dominant-parameter keys for novelty, stall, and seal decisions.
- Advances the blocker state or final-goal state more than it adds another isolated safety proof.
- Minimizes repeated prerequisite work by declaring `current_only` or `affected_chain` validation when full historical reruns are not needed.
- Is narrow enough to become one actionable `task.md`, or narrow enough to become the next item in a bounded task pack.
- Consumes an active task-pack item when that item is still safe, aligned, and not blocked by repeated blocker signatures or missing input delta; otherwise records the pack mutation needed before the next execution item.
- Preserves measurable directive scope with `scope_fidelity`; if the selected item is intentionally narrowed, it keeps the residual target open instead of redefining completion around the smaller slice.
- Preserves measurable directive quantifiers and `evidence_kind`; if the selected item requires live execution, it requires a post-item run id and does not replace it with a derived artifact or report matrix.
- Preserves instrumentation lifecycle ordering; if instrumentation was supplied but not exercised, it selects or inserts exercise before dependent measurement/adoption.
- Preserves measurable verifier completeness: if a measurable target has a required verifier that is `not_evaluated`, the selected task must keep verifier work or residual scope open and cannot consume the target.
- Preserves measurable goal-axis completeness: if review evidence has no observing axis for an active measurable goal, the selected task must keep axis-supply work or residual scope open and cannot consume the target.
