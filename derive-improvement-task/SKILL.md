# Derive Improvement Task

## Overview

Use this skill to turn current goal context, repository evidence, previous misses, and candidate ideas into the next concrete `task.md`.

When long-range ordering is needed, this skill may also create or update a task pack under `.task/task_pack/`. The pack is planning state only; the final executable output remains one active `task.md`. Treat pack updates as a transaction: choose exactly one `pack_disposition`, mutate the JSON queue with evidence, refresh the Markdown render, and promote at most one item into `task.md`.

The workflow is intentionally agent-heavy: use `$inspect-repo-with-agents` for critical goal/convention-alignment analysis, agent-based `.task/task_miss` analysis, one issue goal-fit agent from `$manage-implementation-issues`, three parallel improvement-analysis agents, one synthesis agent, and `$record-agent-work-log` to preserve the previous `task.md` as `past_task` before replacing it.

Use `$manage-task-state-index` for task, candidate, miss, past_task, log, audit, and validation IDs whenever possible. If no ID context exists, continue deriving the improvement task and report that indexing was skipped.

Use `.agent_goal/goal_schema_contract.md` and `.schema/` or `.contract/` contract context from `$manage-schema-contracts` when present. Schema, module, script, version, and causal compatibility contracts are planning evidence, not optional documentation, when the selected task can change interfaces or data artifacts.

Use `$manage-agent-authority` to resolve authority goal truth for task selection. Its result should describe the current agent's effective operating permissions and any narrower project-specific policy for API calls, external service use, direction freedom, strictness, implementation-first versus validation-first priority, output/artifact confirmation, quality review, conservative implementation, and escalation. If the file is absent, consume `authority_policy: default_current_agent_permissions`: current sandbox, approval, user/developer/system instructions, and no inferred extra API/network/destructive authority. This policy may constrain or rank task options, but it cannot override higher-priority instructions or grant missing capabilities.

Use `$manage-external-advice` when `.agent_advice/active/*.md` exists or the caller passes an advice packet. Active advice is non-GT planning evidence: it can steer task choice, design notes, schema planning, and validation expectations, but it must be listed as `used_advice`, kept out of `.agent_goal`, and rejected or deferred if it conflicts with GT, authority, current user instruction, or repository facts.

Use `.issue/` and GitHub issue mirrors from `$manage-implementation-issues` when present. Active issues are planning evidence only after a first-pass issue agent classifies whether each issue fits the goal.

Use `.task/task_pack/*.json` when present. The JSON queue is canonical; `.md` files beside it are user-language renders. Read [task-pack-workflow.md](../orchestrate-task-cycle/references/task-pack-workflow.md) when a caller supplies a task-pack packet or `.task/task_pack/` exists.

If `task.md` is absent, run this skill in initial-init mode: derive the first actionable `task.md` from goal context, repository evidence, issues, task_miss, and candidate_task without requiring or archiving a previous task.

## Domain Adapter Contract

This skill usually consumes adapter-derived fields through caller packets, especially `anti_loop_progress_gate`, `qualitative_review_packet`, validation packets, schema packets, or task-pack metadata. When it directly receives adapter access, it may rely on these optional Part N and Part O hooks without interpreting domain-specific meanings:

- `substance_density(metric_id, **context) -> dict`: N1 helper returning `referent_meaning_ratio`, `opaque_surrogate_ratio`, and adapter-owned `floor`. Missing hook is fail-quiet and preserves existing task selection while recording `substance_density_unchecked` when supplied by the caller.
- `output_fingerprint(...) -> str|dict`: N2 helper returning an opaque deliverable fingerprint and optional freshness metadata. This skill compares only supplied opaque equality/order fields and never builds fingerprint logic.
- `producer_source_paths(artifact_family, **context) -> list|dict`: N2 helper returning producer/verifier source paths or globs for deliverable freshness. If absent, fail quiet and keep existing selection rules.
- `persistence_policy_map(**context) -> dict`: N3 helper returning opaque storage-policy classes, actually persisted classes, leak indicators, and resolution-depth enums. Missing hook is fail-quiet; do not invent field classes or privacy thresholds.
- `hook_demand_threshold(**context) -> int|dict`: N4 threshold for decision-relevant `adapter_hook_demand` rows. The adapter, caller, or loopback registry policy owns the threshold; missing or malformed values fail quiet to existing policy and must not change gate verdicts.
- `landed_feature_inventory(**context) -> dict`: P1 helper returning adapter-owned landed feature inventory and authorized opaque feature classes. This skill consumes only caller/loopback scalar regression fields for selection and never invents feature classes. Missing hook is fail-quiet.
- `producer_execution_evidence(run_id, **context) -> dict`: P2 helper returning whether a run performed fresh producer execution. This skill uses it only when caller/acceptance evidence requires `fresh_producer_execution`; missing hook is fail-quiet unless a caller contract requires the gate.
- `input_source_lineage(run_id, **context) -> dict`: P4 helper returning opaque input lineage and generation-time defect dominance. Missing hook is fail-quiet and preserves existing post-processing selection rules.
- `self_resolvable_input_probe(required_missing_input, **context) -> dict`: Q1 helper returning whether a missing input can be generated by deterministic rewiring of existing repository capabilities without new authority. Missing hook is fail-quiet and preserves existing terminal/escalation selection.
- `reason_to_attempt(reason_code, **context) -> dict`: S3 helper returning the exact attempted `root_key`, `attempt_count`, and optional evidence refs for an opaque fail/degrade reason code. Use it to prove root-cause exhaustion before terminal blocking or sealing. If any exact primary reason has `attempt_count==0`, select that repair before terminal/user escalation. If absent or malformed, fail quiet, preserve existing terminal selection rules, and carry `coverage_unverified_vs_reason_codes`.
- `reason_code_rank(reason_codes, **context) -> dict`: Q4 helper returning `primary`, `secondary`, and `precedence_edges` for fail-close reason codes. The adapter owns reason-code meanings and ordering. Missing hook is fail-quiet and preserves existing selection ranking.
- `code_convention_contract(**context) -> dict`: existing Q6 contract source for purpose directory taxonomy, allowed depth, reuse or abstract roots, and dependency-layer order. Missing contract is fail-quiet for selection; depth/fan-out evidence remains warn-only unless caller packets provide contract-backed or coupled structure findings.
- `repair_file_lineage(changed_paths, blocker_signature, **context) -> dict`: O1 helper returning opaque lineage fields such as `lineage_key`, `same_module_repair_streak`, `repair_shape`, and adapter-owned threshold status. Missing hook is fail-quiet and preserves existing root-cause selection.
- `goal_axis_map(targets=None, quality_vector=None, **context) -> list|dict`: O2 helper returning adapter-owned goal axis ids, delta basis fields, and optional close-time `axis_delta` / `axis_stall_streak` evidence. Missing hook is fail-quiet and preserves existing preparation/direct-work ranking.

Keep Part N distinct from Part M: M routes execution-context exit contracts and reharvest/disposition predicates; N routes landed-output meaning, deliverable freshness, persistence-policy ability/risk alignment, and hook-demand accounting. Do not select two separate tasks for the same underlying failure unless the packets report distinct M and N predicates.

Keep Part O distinct from B/G/N accounting: O1' is same-lineage special-case repair repetition rather than distinct-root novelty, and O2' is aggregate goal-axis stall rather than a claim that any individual cycle was diluted, vacuous, or incorrectly classified. Keep Part P distinct from L/N/O/F accounting: P1 is production-configuration inheritance regression, P2 is evidence freshness class, and P4 is frozen-input marginal value exhaustion. Keep Part Q distinct from L/N/O/P accounting: Q1 changes terminal/escalation routing only when missing input is self-resolvable, Q4 ranks reason-code repairs, and Q6 strengthens code-structure refactor handoff without creating a new regression family.

## Agent Routing Policy

- When called from `$orchestrate-task-cycle`, resolve every delegated role through [model-effort-routing.md](../orchestrate-task-cycle/references/model-effort-routing.md) and its deterministic selector.
- Use Tier 3 balanced-profile/high for goal/convention alignment, task_miss analysis, issue goal-fit, improvement candidates, candidate prioritization inputs, and schema-contract planning. Use Tier 4 balanced-profile/xhigh for cross-contract/blocker-level analysis that recommends but does not own final direction.
- Use Tier 5 direction-profile/xhigh for exactly one final synthesis/selection owner that writes or selects the next `task.md`, chooses task-pack topology/disposition, resolves final GT/authority direction, or emits a terminal disposition.
- Route separate ID-only consistency through `$manage-task-state-index` at Tier 2 balanced-profile/medium; do not spend derive synthesis effort on traceability-only work.
- Use Tier 5 direction-profile/max only for exactly one bounded arbitration after a prior Tier 5 direction-profile/xhigh synthesis leaves a documented high-impact ambiguity involving conflicting GT/authority contracts, a security-critical irreversible direction, or terminal disposition. Record `prior_tier5_unresolved: true`, `prior_tier5_evidence`, `max_escalation_reason`, and `agent_count: 1`.
- If a called skill owns subagents, include the matching tier profile in the request. Never use delegated `ultra`.
- If the delegation tool cannot enforce model/effort, preserve the tier/profile/model-reference request in the prompt and report `routing_enforcement: prompt_only|inherited_unverified` plus the limitation. Do not claim configured-model execution.
- If the thread/agent budget cannot run the requested fanout, preserve independent lenses where possible, keep synthesis evidence-separated, and record `degraded_agents: true` with unavailable agents, covered/missing lenses, and risk.

## Workflow

Before ordinary derivation, require the predecessor's verified current finalization receipt, its exact authoritative projection, and a `finalization_consumption` echo of the receipt token, attempt/revision, projection ID/digest, and receipt hash. Reject a missing, stale, superseded, malformed, or mismatched receipt before reading predecessor conclusions, promoting a pack item, writing `task.md`, or selecting terminal/user handoff. Consume task acceptance and goal readiness as separate axes; a task-level pass with semantic or goal failure cannot become `goal_productive`.

Skip this receipt gate only for `initial_init`/`bootstrap`, or for an explicit `finalization_applicability: not_applicable` transition whose `prior_final_attempt_exists` is false, whose reason is a bounded opaque ID, and whose transition kind is `standalone_repair`, `unrelated_state_repair`, or `no_predecessor_attempt`. Pending advice, a new task/family label, or a changed timestamp does not qualify for this exception and cannot change authority or goal truth.

Consume loopback `authoritative_semantic_progress` and hard-stop facts monotonically. Never upgrade false/block to true/advanced from review wording, report fields, dashboard state, or terminal-outcome labels. If required actual-body truth, report convergence, consumer context, artifact class, freshness/current lane, or verifier completeness is `not_evaluated`, do not select an `advanced` interpretation.

Optionally consume only a trusted-collector privacy-safe projection from [$audit-session-governance](../audit-session-governance/SKILL.md), never raw/slim transcript text or a self-promoted result packet. A separate deterministic comparator receipt may constrain selection or propose an issue/repair task; a session-owned relation claim cannot establish authority, completion, progress, blocker truth, or positive input delta. Treat capture absence, quarantine, incomplete capture, and transcript-only observations as advisory/`not_evaluated` unless acceptance or the caller independently requires the audit. Do not inline a stop-hook repair: route source, task, acceptance, goal, and authority changes through their owning skills; only deterministic audit-index reconstruction is unattended repair, and it retains original evidence.

Before choosing terminal/user escalation, consume the loopback residual classification. `self_resolvable_local`, `offline_recompute`, and `existing_authority` require a bounded local task; `unverified` requires classification/probe work. Only genuine new authority or external state may support external escalation after existing-authority work is exhausted. `pack_exhausted_needs_derivation` is not goal terminal.

When escalation is applicable, emit `option_inventory` version 1 with a complete list of currently applicable or evidenced-absent options, unique opaque option/evidence IDs, typed option class/applicability, and a derived `blocker_removing_option_present`. Do not require every option class merely for shape. If no blocker-removing option is applicable, bind the absence reason to evidence IDs. An incomplete or `not_evaluated` inventory may request information but cannot establish terminal, completion, or authority truth.

Before consuming a gate for an operation, require adapter/authority-supplied `gate_operation_applicability` version 1 and name `requested_operation`. Unknown scope blocks publish, promote/adopt, offline state-changing transform, and destructive disposition. A diagnostic read is allowed only when its matrix row is allowed and a separate content-bound receipt verifies authority, safety, privacy, and provenance. Operation applicability never expands the current authority envelope.

When family profile scope is unverified, ignore global any/max exhaustion and hard-stop aggregates. Use only the current goal axis, stable root family, producer lineage, artifact class, decision lane, and input cohort. A trace-label rename cannot create a new family. When the existing sprawl hard gate is active, select consolidation, goal-productive producer work, terminal block, or escalation as allowed; do not merely report it.

Do not drop a required structure-repair lineage because the debt is pre-existing or because an outer packet declares the slice complete. Until adapter-owned structure high-water moves, select the in-place repair, explicit residual/descope, terminal block, or escalation already allowed by the existing disposition contract.

## Selection Rules

Prefer the improvement that:

- Best advances `.agent_goal/final_goal.md` success criteria.
- Respects `.agent_goal/conventions.md`; if a convention conflict exists, the selected task explicitly resolves or documents it as a blocker rather than proceeding as if compliant.
- Fits the documented architecture and theory, or explicitly updates gaps discovered by analysis.
- Incorporates active `.agent_advice` only when it is compatible with current user direction, GT, authority, and repository facts; otherwise explicitly defers or rejects it.
- Preserves documented `.schema` and `.contract` compatibility, or makes the necessary schema/contract version update explicit in the task.
- Respects the `$manage-agent-authority` result, including `.agent_goal/agent_authority.md` when present or the `default_current_agent_permissions` fallback when absent, including API/external-call policy, direction freedom, strictness, implementation/validation priority, and conservative implementation constraints.
- Satisfies `.agent_goal/goal_schema_contract.md` minimum fields, mandatory rules, target module/script guidance, and causal-map expectations when schema/contract surfaces are involved.
- Addresses high-severity `.task/task_miss` patterns.
- Fits active implementation issues that the issue agent classifies as goal-aligned, or explicitly explains why a misfit/stale/duplicate issue should not drive the next task.
- Can be validated with concrete tests, commands, metrics, or review evidence.
- Is `goal_productive` when the caller's goal-distance gate requires it: expected to produce changed-and-semantic goal-relevant output, quality evidence, source-backed validation, or another non-sidecar artifact that reduces goal distance.
- Uses a `selected_task_kind` allowed by `disposition_intersection_basis.allowed_task_kinds` whenever gates constrain `goal_productive` to specific correction classes.
- Is not classified as `goal_productive` merely because it writes workflow metadata, sidecar metrics, schema records, task-state records, non-empty counts, lineage, gap reports, or measured absences without changed-and-semantic repository-declared output delta or validated positive evidence.
- Does not consume `pass_with_coupled_verifier`, `producer_attested_fields`, or `attested_only_movement` as progress truth.
- Does not consume contradictory terminal classifications, same-input mismatches, diagnostics-unavailable repeats, non-disjoint independent verification, or missing envelope thaw items as ordinary `goal_productive` progress.
- Does not consume supplied-but-unexercised instrumentation, uncovered scenarios, acceptance inversion, missing command provenance, repeated opaque blockers, stochastic infeasible contracts, first-fire instrumentation credit, live-run criteria satisfied by derived/report/code-contract evidence, guard-stacking hardening past cap, or `candidate_degraded` artifacts as ordinary `goal_productive` progress.
- Does not consume stale output-derived expectations, unverified comparison parity, failed gating adoption axes, lower-resolution surrogate evidence, or divergent duplicate report keys as ordinary `goal_productive`, close, baseline, comparison, adoption, or high-water evidence.
- Does not consume stale-lane passes, stale decision measurements, producer-starved gating axes, restrictive verifier-like portfolio excess, cycle-unreachable smoke reruns, basis-overclaimed metrics, or unresolved surface-field defect counts as ordinary `goal_productive`, close, baseline, comparison, adoption, next-rung, or high-water evidence.
- Does not consume risky harvest preflight, destructive high-cost non-safety disposition, rerun-before-reharvest, mutually unsatisfiable predicate/directive contracts, or truncated/sample collections as ordinary `goal_productive`, close, baseline, comparison, adoption, next-rung, or high-water evidence.
- Does not consume `pass_with_unobserved_axes` or a failed `goal_axis_completeness_gate` as qualitative pass evidence for measurable goals.
- Treats below-threshold residual gap repair as `marginal_repair` unless adapter evidence shows its marginal value, including value per cycle cost when supplied, exceeds explicit residual descope and the next capability rung.
- Treats generation-dependent family/root keys as trace-only and uses effective root-family/dominant-parameter keys for novelty, stall, and seal decisions.
- Advances the blocker state or final-goal state more than it adds another isolated safety proof.
- Minimizes repeated prerequisite work by declaring `current_only` or `affected_chain` validation when full historical reruns are not needed.
- Is narrow enough to become one actionable `task.md`, or narrow enough to become the next item in a bounded task pack.
- Consumes an active task-pack item when that item is still safe, aligned, and not blocked by repeated blocker signatures or missing input delta; otherwise records the pack mutation needed before the next execution item.
- Preserves measurable directive scope with `scope_fidelity`; if the selected item is intentionally narrowed, it keeps the residual target open instead of redefining completion around the smaller slice.
- Preserves measurable directive quantifiers and `evidence_kind`; if the selected item requires live execution, it requires a post-item run id and does not replace it with a derived artifact or report matrix.
- Preserves instrumentation lifecycle ordering; if instrumentation was supplied but not exercised, it selects or inserts exercise before dependent measurement/adoption.
- Preserves measurable verifier completeness: if a measurable target has a required verifier that is `not_evaluated`, the selected task must keep verifier work or residual scope open and cannot consume the target.
- Preserves measurable goal-axis completeness: if review evidence has no observing axis for an active measurable goal, the selected task must keep axis-supply work or residual scope open and cannot consume the target.

## Static Decision-Boundary Fixtures

- Starvation negative: `body_fp_A` is unchanged and an evaluated portfolio budget is reached by verifier/report/structure items. Expected verdict: rank producer/body repair or bounded producer execution ahead of another verifier-like item. Forbidden overclaim: treating a renamed check or receipt repair as semantic movement.
- Starvation happy path: approved `attempt_A` produces a fresh exact-bound body and independently verified `axis_G` movement. Expected verdict: producer-starvation routing clears for that path. Forbidden overclaim: inferring completion or movement on other axes.
- Lifecycle negative: `task_T` has bounded acceptance `pass`, global readiness `blocked`, no successor, and an unchanged mutable alias. Expected verdict: do not select `task_T` again; keep it non-executable. Forbidden overclaim: equating the alias's presence with an active executable task.
- Lifecycle happy path: a distinct `task_U` is later selected with its own canonical identity while `task_T` remains immutable history. Expected verdict: only `task_U` is executable. Forbidden overclaim: rewriting or reactivating `task_T`.
